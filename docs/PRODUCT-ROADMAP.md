## 文档概述

本文档是 Seahorse Agent 平台的中长期产品演进规划，基于**纯云端 SaaS 部署**定位，聚焦多租户、商业变现、自动化运维等 SaaS 核心能力。

**目标受众**：产品经理、技术负责人、开发团队  
**更新频率**：季度评审  
**产品定位**：企业级 RAG 智能体 SaaS 平台  

**优先级定义**：
- **P0（关键）**：影响商业闭环、数据安全或服务稳定性
- **P1（高）**：显著提升用户体验或运营效率
- **P2（中）**：差异化竞争的高级特性
- **P3（低）**：锦上添花的优化

## 演进策略

### 四阶段路线图（云端 SaaS 优化版）

```
2026 Q2-Q3: MVP 上线期 → 2026 Q4-2027 Q1: 规模化运营期 → 2027 Q2-Q3: 功能增强期 → 2027 Q4+: 生态建设期
```

### 核心原则

1. **商业闭环优先**：Phase 1 必须完成计费系统，实现收入
2. **数据安全第一**：Phase 1 必须完成多租户隔离
3. **服务质量保障**：Phase 1 必须完成基础监控告警
4. **快速转化**：Phase 1 必须支持自助注册和免费试用
5. **删除非必要复杂度**：云端固定技术栈，不暴露 Adapter 切换

### 投入分配建议

- **MVP 上线期**：50% SaaS 基础设施，30% 核心功能完善，20% 体验优化
- **规模化运营期**：40% 运维自动化，30% 用户运营，30% 功能完善
- **功能增强期**：50% 差异化功能，30% 企业能力，20% 优化
- **生态建设期**：60% 开放平台，25% 高级 AI 特性，15% 国际化

### 与传统路线图的关键差异

| 维度 | 本地部署版 | 云端 SaaS 版 |
|------|-----------|-------------|
| 多租户 | Phase 3（可选） | **Phase 1（必需）** |
| 计费系统 | Phase 4（可选） | **Phase 1（必需）** |
| 用户注册 | 不需要 | **Phase 1（必需）** |
| 运维监控 | Phase 2 | **Phase 1（必需）** |
| Adapter 切换 | Phase 1 | **删除（云端固定）** |
| 本地部署文档 | 核心 | **删除（纯云端）** |
| 自动扩缩容 | 不需要 | **Phase 2（必需）** |
| 在线客服 | 不需要 | **Phase 2（必需）** |

## Phase 1: MVP 上线期（2026 Q2-Q3，4-5 个月）

**目标**：支持首批 100 个付费用户上线，实现商业闭环。

**验收标准**：
- 支持 100 个并发租户，数据完全隔离
- 支付成功率 ≥ 99%
- 系统可用性 ≥ 99%
- 用户注册到首次使用 < 5 分钟

### 1.1 用户体系与认证（P0，新增/增强，2 周）

**现状**：有基础登录，但无自助注册、无免费试用。

**改进方案**：

#### 1.1.1 用户注册与登录

- 注册方式：
  - 邮箱注册（验证码）
  - 手机号注册（短信验证码）
  - 第三方登录（微信、钉钉、GitHub OAuth）
- 注册流程优化：
  - 表单简化（只要邮箱和密码）
  - 自动创建个人租户
  - 自动激活 14 天免费试用
- 登录体验：
  - 记住登录状态（7 天）
  - 多设备登录管理
  - 登录日志和异常检测

**技术实现**：
```java
// 后端用户注册
@PostMapping("/api/auth/register")
public RegisterResponse register(@RequestBody RegisterRequest request) {
    // 1. 创建用户
    User user = userService.createUser(request);
    // 2. 自动创建租户
    Tenant tenant = tenantService.createPersonalTenant(user);
    // 3. 激活免费试用
    trialService.activateTrial(tenant, Duration.ofDays(14));
    // 4. 返回 JWT Token
    return new RegisterResponse(jwtService.generateToken(user));
}
```

### 1.2 多租户架构（P0，前置，3 周）

**现状**：无租户隔离，所有用户数据混合存储，存在严重数据安全风险。

**改进方案**：

#### 1.2.1 数据隔离

- Row-Level Security（行级安全）：
  - 所有业务表增加 `tenant_id` 字段
  - PostgreSQL RLS 策略自动过滤
  - Hibernate Filter 自动注入租户条件
- 租户上下文管理：
  - 请求进入时从 JWT 解析租户 ID
  - ThreadLocal 存储当前租户
  - 所有查询自动带租户过滤
- 跨租户查询防护：
  - 禁止直接 SQL 查询
  - 所有查询必须经过 Repository 层
  - 自动化测试验证隔离性

**技术实现**：
```java
// 租户上下文
@Component
public class TenantContext {
    private static final ThreadLocal<String> currentTenant = new ThreadLocal<>();
    
    public static void setTenant(String tenantId) {
        currentTenant.set(tenantId);
    }
    
    public static String getTenant() {
        String tenantId = currentTenant.get();
        if (tenantId == null) {
            throw new IllegalStateException("Tenant context not set");
        }
        return tenantId;
    }
}

// 拦截器自动设置租户
@Component
public class TenantInterceptor implements HandlerInterceptor {
    @Override
    public boolean preHandle(HttpServletRequest request, ...) {
        String tenantId = jwtService.extractTenantId(request);
        TenantContext.setTenant(tenantId);
        return true;
    }
}

// JPA 自动过滤
@FilterDef(name = "tenantFilter", parameters = @ParamDef(name = "tenantId", type = "string"))
@Filter(name = "tenantFilter", condition = "tenant_id = :tenantId")
@Entity
public class KnowledgeBase {
    @Column(name = "tenant_id", nullable = false)
    private String tenantId;
}
```

**工作量**：2 人 × 10 天

### 1.3 基础计费系统（P0，前置，3 周）

**现状**：无计费系统，无法商业化。

**改进方案**：

#### 1.3.1 套餐管理

- 三档套餐：
  - **基础版**（¥99/月）：存储 10GB、Token 100 万/月、并发 5、基础功能
  - **专业版**（¥499/月）：存储 50GB、Token 500 万/月、并发 20、高级功能（Agent 编排、自动评测）
  - **企业版**（定制）：无限制、专属部署、SLA 保障、技术支持
- 按量付费选项：
  - 超出套餐的 Token：¥0.01/1K tokens
  - 超出套餐的存储：¥0.1/GB/月
- 年付优惠：
  - 年付享 8 折优惠

**技术实现**：
```java
// 套餐定义
public enum SubscriptionPlan {
    FREE_TRIAL(0, 5_000_000_000L, 50_000_000L, 3),  // 5GB, 50万 tokens
    BASIC(99, 10_000_000_000L, 100_000_000L, 5),    // 10GB, 100万 tokens
    PRO(499, 50_000_000_000L, 500_000_000L, 20),    // 50GB, 500万 tokens
    ENTERPRISE(-1, -1L, -1L, -1);                    // 无限制
    
    private final int monthlyPrice;
    private final long storageQuota;
    private final long tokenQuota;
    private final int concurrencyQuota;
}

// 配额检查
@Service
public class QuotaEnforcer {
    public void checkStorageQuota(String tenantId, long fileSize) {
        Subscription sub = subscriptionService.getActiveSub(tenantId);
        long used = usageService.getStorageUsed(tenantId);
        if (used + fileSize > sub.getStorageQuota()) {
            throw new QuotaExceededException("存储空间不足，当前已使用 " + formatSize(used) + "，请升级套餐");
        }
    }
}
```

## Phase 2: 规模化运营期（2026 Q4-2027 Q1，4 个月）

**目标**：支撑 1000+ 付费用户，建立自动化运营体系。

**验收标准**：
- 支持 1000+ 并发租户
- 系统可用性 ≥ 99.5%
- 客户流失率 < 10%
- MTTR（平均修复时间）< 30 分钟

### 2.1 完善运维监控（P0，2 周）

#### 2.1.1 统一运维仪表板

- 实时指标展示：
  - 系统健康度、活跃用户、QPS、错误率
  - 折线图：最近 24 小时的趋势
  - 依赖服务状态表格
- 业务监控：
  - Agent 执行次数和成功率
  - 知识库检索延迟分布
  - Token 消耗趋势
  - 收入趋势（MRR、ARR）

**工作量**：2 人 × 7 天

#### 2.1.2 告警规则引擎

- 自定义告警规则：
  - 支持阈值告警（指标 > 阈值）
  - 支持同比/环比告警（增长率异常）
  - 支持复合条件（AND/OR）
- 告警分级和通知：
  - P0（紧急）→ 电话 + 钉钉 + 邮件
  - P1（重要）→ 钉钉 + 邮件
  - P2（一般）→ 邮件
- 告警处理工作流：
  - 新建 → 已确认 → 处理中 → 已解决 → 已关闭

**工作量**：1 人 × 5 天

### 2.2 自动化运维（P0，3 周）

#### 2.2.1 自动扩缩容

- 水平扩缩容（Kubernetes HPA）：
  - CPU 使用率 > 70% 时自动扩容
  - CPU 使用率 < 30% 时自动缩容
  - 最小副本数 3，最大副本数 20
- 数据库连接池动态调整：
  - 根据并发请求数动态调整连接池大小
- 成本优化：
  - 低峰期（凌晨 2-6 点）自动缩容到最小副本
  - 按需启动工作节点（Spot 实例）

**工作量**：1 人 × 5 天

#### 2.2.2 服务降级开关

- 降级策略：
  - 高峰期关闭非核心功能（统计报表、历史查询）
  - 限制 Agent Run 并发数
  - 限制文档上传大小和频率
- 降级开关：
  - 运维后台一键开启/关闭
  - 自动检测负载并降级（CPU > 90%）
- 用户提示：
  - 降级时友好提示："系统繁忙，部分功能暂时不可用"

**工作量**：1 人 × 3 天

#### 2.2.3 数据备份与恢复

- 自动备份：
  - 数据库每天全量备份（凌晨 3 点）
  - 向量索引每周备份
  - 对象存储增量备份
- 备份保留策略：
  - 最近 7 天：每日备份
  - 最近 4 周：每周备份
  - 最近 12 个月：每月备份
- 恢复演练：
  - 每月一次恢复演练
  - 恢复时间目标（RTO）< 1 小时

**工作量**：1 人 × 5 天

### 2.3 用户运营体系（P1，新增，3 周）

## 风险与挑战

### 技术风险

1. **多租户隔离风险**：数据泄露可能导致严重后果
   - 缓解：严格代码审查、自动化测试、定期渗透测试

2. **性能瓶颈**：大规模并发时向量检索可能成为瓶颈
   - 缓解：性能测试、索引优化、水平扩展

3. **成本失控**：AI 模型调用成本可能快速增长
   - 缓解：智能缓存、成本监控告警、Prompt 优化

### 商业风险

1. **免费试用滥用**：用户反复注册获取免费额度
   - 缓解：手机号验证、IP 限制、行为检测

2. **客户流失**：竞争激烈可能导致客户流失
   - 缓解：差异化功能、优质服务、用户运营

3. **合规风险**：GDPR、数据安全法等合规要求
   - 缓解：提前规划合规能力、聘请法律顾问
