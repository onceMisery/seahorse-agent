# Seahorse Agent 云端 SaaS MVP 执行路线图

日期：2026-06-04
状态：可排期执行版（v2：含现有功能成熟度审查 + 加固 + 未来期望）

## 本文定位

这是一份**工程执行视角**的路线图，回答"Skills 做完之后该怎么规划"。它包含两条主线：

- **主线一 · SaaS 新建能力（块A–E）**：把现有单体平台推上云端 SaaS MVP（多租户、用户、计费、监控）。
- **主线二 · 现有功能加固与闭环（域1–6）**：基于对六大功能域的逐域代码审查，列出成熟度、闭环缺口、加固项和未来期望。

两条主线**可并行**：主线一是"从无到有"建 SaaS 地基，主线二是"从能用到好用"打磨已有能力。文末有统一的优先级合并视图。

它与 `docs/PRODUCT-ROADMAP.md` 的关系：产品路线图给的是商业目标和阶段划分；本文基于 **2026-06-04 的实际代码审查**，修正了产品路线图里几处过时的现状评估，并把工作拆成可直接分工、带依赖顺序和验收信号的工作块。

**贯穿全文的核心发现：后端普遍比前端成熟。** 六大域审查反复出现同一模式——后端接口/领域模型已实现且有测试，但前端缺操作入口，导致能力"存在但用户摸不到"。因此主线二的最大一类工作不是"造新功能"，而是**把已有后端能力补上前端闭环**。这类工作单价低、收益高、风险小，适合穿插在 SaaS 主线的间隙做。

**两处关键修正（直接影响工作量，务必先读）：**

| 维度  | 产品路线图旧评估             | 代码审查真实现状                                                                                   | 影响                                      |
| --- | -------------------- | ------------------------------------------------------------------------------------------ | --------------------------------------- |
| 计费  | 2.0/10，"无计费系统，无法商业化" | 配额 + 用量统计底座已 ~80%（`sa_quota_policy`、`sa_cost_usage_record`，可按租户/用户/Agent/Run 统计 token 与成本） | 计费 MVP 工作量大幅下降，只需补套餐/支付/账单层             |
| 多租户 | 3.0/10，"无租户隔离，数据混合"  | 81% 表已有 `tenant_id` 字段；缺的是强制隔离机制，不是字段                                                      | 工作重点是 TenantContext + JWT + 过滤审计，而非全表改造 |

## 关键洞察：依赖顺序决定排期

这是规划的核心。四个 P0 工作块**不是平行的**，存在硬依赖：

```
        ┌──────────────────────────────┐
        │  块A 多租户隔离（地基）        │  ← 必须最先做
        │  TenantContext + JWT tenantId │
        └──────────────┬───────────────┘
                       │ 所有按租户隔离/计费的逻辑都依赖它
        ┌──────────────┼──────────────┐
        ▼              ▼              ▼
   ┌─────────┐   ┌─────────┐   ┌─────────┐
   │块B 用户  │   │块C 计费  │   │（块A 完成后）│
   │注册建租户│   │配额按租户│   │            │
   │+ 试用    │   │强制+套餐 │   │            │
   └─────────┘   └─────────┘   └─────────┘

   ┌────────────────────────────────────┐
   │ 块D 运维监控（独立，可全程并行）     │  ← 不依赖多租户，最先并行启动
   └────────────────────────────────────┘
```

**结论：**

- **块A（多租户）是地基**，块B、块C 都依赖它的 TenantContext。先做 A，否则 B/C 返工。
- **块D（监控）完全独立**，应在第一周就并行启动，不占关键路径。
- **块E（Skills 收尾）是轻量收尾**，1 人周内完成，可穿插。

### 块A：多租户隔离强制化（地基，关键路径，2 人 × 8–10 天）

**现状**：81% 表有 `tenant_id`，但无强制机制——开发者漏传一个参数就跨租户泄露。这是 SaaS 最高优先级的数据安全前置。

**做什么：**

- [ ] `TenantContext`（ThreadLocal）+ `TenantInterceptor`：请求进入时从 JWT 解析 tenantId 并入栈，请求结束清理。
- [ ] JWT/Sa-Token 登录时写入 tenantId，鉴权链路解析。
- [ ] **审计现有 Repository 查询**：列出所有 JDBC 查询，确认每条都带 `tenant_id` 过滤；补齐遗漏的（这是真正的工作量大头，不是加字段）。
- [ ] 补齐尚未带 `tenant_id` 的核心表（用户表、KB、会话等约 18 张，从调研结论取准确清单）。
- [ ] PostgreSQL RLS 策略作为**第二道防线**（`ENABLE ROW LEVEL SECURITY` + `CREATE POLICY`），即使应用层漏过滤，DB 层也兜底。
- [ ] 跨租户隔离自动化测试：构造租户 A/B 数据，断言 A 永远查不到 B。

**不做**：多租户的计量/计费逻辑（归块C）；租户管理后台 UI（Phase 2）。

**依赖**：无（地基）。**注意**：新增的租户 autoconfig 要遵循 CLAUDE.md 的 6 层 `@AutoConfigureAfter` 规则。

**验收**：隔离测试全绿；关闭应用层过滤后 RLS 仍拦截跨租户；JWT 不含 tenantId 的请求被拒。

---

### 块B：用户体系（注册 / 试用 / RBAC，1–2 人 × 6–8 天）

**现状**：Sa-Token 框架完整，但只有基础登录，无自助注册闭环。

**做什么：**

- [ ] 邮箱注册 + 验证码；表单简化（邮箱 + 密码）。
- [ ] 注册即**自动创建个人租户**（依赖块A 的租户模型）+ 激活 14 天试用。
- [ ] 试用期管理：配额绑定（存储/Token/并发）、到期提醒、到期降只读。
- [ ] 记住登录（7 天）。

**不做 / 后延 Phase 2**：第三方 OAuth（微信/钉钉/GitHub）、细粒度 RBAC、多设备管理、SSO。MVP 用现有 Sa-Token 角色即可。

**依赖**：块A（自动建租户需要租户模型）。

**验收**：新用户邮箱注册 → 自动建租户 → 14 天试用激活 → 5 分钟内可用。

---

### 块C：计费系统（复用配额底座，2 人 × 8–10 天）

**现状**：配额 + 用量统计已 ~80%。这是最被低估的优势——**不用从零建**。

**做什么：**

- [ ] 套餐定义（`SubscriptionPlan`：免费试用/基础/专业/企业），绑定到已有 `QuotaPolicy`。
- [ ] 把已有的 `/api/quotas/decisions:evaluate` **接入运行时强制**：上传文档前查存储配额、执行 Agent 前查 Token 配额。
- [ ] 用量自动聚合任务（现在 `CostUsageRecord` 需主动调 aggregate，补定时聚合）。
- [ ] 支付集成（支付宝/微信扫码 + H5）：订单状态机、回调幂等、签名验证。
- [ ] 账单生成（每月 1 号生成上月账单）+ 历史查询。

**不做 / 后延**：发票/增值税专票、按量付费精算、年付优惠（Phase 2）。

**依赖**：块A（配额按 tenant 维度强制）。**安全**：支付回调必须幂等 + 签名校验，这是高风险点。

**验收**：选套餐 → 下单 → 支付 → 回调激活 → 配额生效；超配额时友好拦截并引导升级；支付成功率 ≥ 99%。

---

### 块D：运维监控（独立并行，1 人 × 8–10 天）

**现状**：有 SRE Health 框架（5 contributor）+ Micrometer，但缺 Actuator、缺中间件直连检查、缺告警、缺仪表板。

**做什么：**

- [ ] 引入 `spring-boot-starter-actuator` + `micrometer-registry-prometheus`，暴露 `/actuator/prometheus`。
- [ ] 补 DB/Redis/Milvus/ES 直连 HealthContributor（现有 5 个偏适配器层，缺中间件连通性）。
- [ ] 告警：钉钉群机器人 webhook，内置规则（服务 Down、错误率 >5%、连接池耗尽、支付回调失败、配额耗尽）。
- [ ] Grafana 仪表板：系统健康、QPS、错误率、活跃租户、Token 消耗、收入趋势。

**不做 / 后延**：自定义告警规则引擎、灰度流量分流（现仅有 `AgentVersionRollout` 框架，缺分流逻辑，Phase 2）、自动扩缩容。

**依赖**：无（全程可并行，建议第一周就启动）。

**验收**：`/actuator/prometheus` 有指标；Grafana 能看趋势；触发一次模拟故障，钉钉群收到告警。

# 主线二 · 现有功能加固与闭环

主线一是"建 SaaS 地基"。主线二是"把已有的六大功能域从能用打磨到好用"。

下面每个域给四样东西：**成熟度矩阵**（子能力逐个评级）、**闭环缺口**（后端有/前端无的清单）、**加固项**（P0/P1/P2）、**未来期望**。评级口径统一为：

- **成熟**：功能完整 + 有测试 + 前后端闭环 + 有错误处理
- **可用**：核心路径通，但缺边界/测试/部分闭环
- **粗糙**：能跑但明显不完整
- **缺失**：占位或无

## 域总览（先看这张）

| 功能域               | 整体水平           | 后端                   | 前端                        | 最大短板                  |
| ----------------- | -------------- | -------------------- | ------------------------- | --------------------- |
| 域1 知识库 & RAG 检索   | 可用偏成熟          | 成熟                   | 高级配置缺入口                   | rerank/混合权重/评测样本无前端   |
| 域2 Agent 运行时 & 编排 | Beta（核心成熟，高阶缺） | 成熟                   | Handoff/ContextPack 几乎无前端 | 多 Agent 协作不闭环         |
| 域3 对话 / Chat 链路   | 生产可用           | 流式/会话成熟，查询理解粗糙       | 体验成熟                      | 查询改写无语义、意图识别退化        |
| 域4 记忆系统           | 可用偏成熟          | 成熟                   | 治理有前端                     | Correction 事务、细化去重未完成 |
| 域5 工具 & 安全 & 治理   | 分层不均           | 网关/ACL/审计成熟，密钥/连接器粗糙 | 多处缺入口                     | 密钥脱敏、ACL 强制、数据完整性缺失   |
| 域6 数据完整性          | 缺失             | 无软删除/回收站/备份          | 无                         | 整块从零                  |

**一句话：核心链路普遍成熟，问题集中在「高阶能力没前端入口」和「安全/治理的几处硬缺口」。**

## 域1：知识库 & RAG 检索

| 向量化（Milvus/pgvector/noop）    | 成熟  | ✅   | 维度无显式配置              |

**加固项：**

- **P0** 关键词索引重建前端按钮（后端 `rebuildKbKeywordIndex` 已就绪，前端接 1 个按钮 + 确认弹窗）。

- **P1** RAG 评测前端闭环：数据集列表/样本编辑/运行/版本对比，挂到 `/admin/rag-evaluation`。
- **P1** 检索策略模板维护页（`retrieval-strategy-templates` 后端 CRUD 已存在）。
- **P2** KB 维度显式配置 + 版本历史追踪 UI。

**未来期望：** 自动质量检测（低质/重复/过时内容）、检索策略 A/B、多模态知识库（图表/音视频）、知识图谱。

## 域2：Agent 运行时 & 编排

**整体：Beta。Agent 运行框架成熟（状态机 + 调试界面），但多 Agent 协作（Handoff）和上下文透明（ContextPack）几乎没有前端，是最突出的不闭环。**

| 子能力                           | 成熟度 | 闭环  | 关键缺口                                           |
| ----------------------------- | --- | --- | ---------------------------------------------- |
| Agent 运行（run/step/状态机/详情）     | 成熟  | ✅   | —                                              |
| Rollout 灰度（时间线可视化）            | 可用  | ✅   | 缺真实流量分流逻辑（仅 canary% 框架）                        |
| Agent 工厂（模板/from-template）    | 可用  | ✅   | —                                              |
| Agent 定义（版本/发布/校验）            | 可用  | 🟡  | 发布校验规则过简                                       |
| Checkpoint 与恢复                | 可用  | 🟡  | 后端完整，前端缺恢复 UI                                  |
| Artifact（详情/preview/download） | 可用  | 🟡  | previewText 未生成、下载入口缺                          |
| **Handoff（多 Agent 转交）**       | 粗糙  | ❌   | 后端完整 + 有测试，前端无 `agentHandoffService`、无创建/编排 UI |
| **ContextPack（运行时上下文）**       | 粗糙  | ❌   | 后端完整，前端有 `contextPackService` 但缺页面入口（看不到注入了什么） |

**加固项：**

# 两条主线的统一优先级视图

把主线一（SaaS 块A–E）和主线二（域加固）的 P0 合在一起，按"该不该进 Phase 1"分三档：

### 第一档 · Phase 1 必做（商业上线阻塞）

| 项                               | 来源  | 为什么阻塞                        |
| ------------------------------- | --- | ---------------------------- |
| 块A 多租户隔离                        | 主线一 | SaaS 数据安全地基，不做则不能多租户         |
| 块B 用户注册/试用                      | 主线一 | 没有自助注册无法获客                   |
| 块C 计费（复用配额底座）                   | 主线一 | 不收费无法商业化                     |
| 块D 运维监控                         | 主线一 | 无监控无法保障 SLA                  |
| 域5 密钥脱敏 / ACL 强制 / 沙箱隔离 / 连接器凭证 | 主线二 | **安全硬缺口，SaaS 多租户下放大为数据泄露风险** |
| 域4 记忆 × 多租户联动                   | 主线二 | 与块A 协同，否则跨租户记忆混淆             |

> 注意：主线二域5 的四个安全 P0 **必须和块A 多租户一起做**。单体时它们是"待改进"，多租户后它们是"数据泄露入口"。这是把两条主线绑在 Phase 1 的关键理由。

### 第二档 · Phase 1.5 强烈建议（体验/闭环，低成本高收益）

这些大多是"后端已有、补前端入口"的闭环工作，单价低：

- 域1：关键词索引重建按钮、rerank/混合权重配置、RAG 评测前端。
- 域2：Handoff 前端闭环、ContextPack 前端闭环。
- 域3：附件失败反馈、Skill 执行可视化。
- 域4：Correction 事务、细化去重。
- 域5：审批重审、前端补缺（ACL/连接器/配额看板）。
- 域6：软删除回收站。

### 第三档 · Phase 2+ 差异化（未来期望）

可视化编排、查询语义化、多模态知识库、工具市场、思维链可视化、多通道、政策即代码等——这些是拉开竞品差距的，但不阻塞上线。

## 需要你拍板的决策点

这些会改变实现方向，建议尽早定：

**主线一（SaaS）：**

1. **团队人力**：上面哪个口径贴近实际？这决定排期能不能落地。
2. **多租户技术选型**：应用层过滤为主 + RLS 兜底（推荐，双保险），还是只做应用层（快但风险高）？
3. **支付渠道**：MVP 只接一个（支付宝 or 微信）先跑通，还是两个都要？
4. **OAuth 第三方登录**：Phase 1 必需，还是可后延到 Phase 2？（邮箱注册已能跑通商业闭环）
5. **试用配额数值**：14 天 / 存储 5GB / Token 50 万/月 / 并发 3 是否合适？

**主线二（加固）：**
6. **安全 P0 是否绑进 Phase 1**：推荐"是"——域5 的密钥/ACL/沙箱/连接器四项在多租户下是数据泄露入口，应与块A 同期做。你认可这个绑定吗？
7. **闭环补缺的节奏**：第二档（Handoff/ContextPack/RAG 评测等前端入口）是穿插进 Phase 1 间隙，还是单列一个 Phase 1.5 集中做？
8. **查询理解投入**：域3 查询改写语义化，先做轻量分词版（2 天），还是直接接小模型（5 天）？
