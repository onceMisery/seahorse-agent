创建日期：2026-05-27

## 1. 分析背景

本文档基于对 5 个开源 Agent 项目的深度源码分析，提炼出对 Seahorse Agent 有实际借鉴价值的技术亮点。本文档定位为**外部项目能力吸收池 + 适配决策**，不直接作为工程排期主文档。工程排期以 `docs/company-agent/c-web-ai-infra-phases/` 为主线。

**产品定位**：Seahorse 当前主线是面向 C 端 Web 场景的云端 Agent 产品。企业级平台能力（Agent mesh、A2A、任意 MCP、工作流 Studio）只作为高级扩展边界，不作为当前 C 端 AI Infra 完成标准。

**分析对象**：

| 项目 | 定位 | 核心价值 |
|------|------|---------|
| bytedance/deer-flow | 深度研究 Super Agent Harness | 上下文工程、子 Agent 编排、渐进式技能加载、MCP 会话池 |
| CopilotKit/CopilotKit | 前端 Agent 原生应用框架 | 事件分族协议、双向状态同步、生成式 UI、Human-in-the-Loop |
| stackblitz/bolt.new | 浏览器内 AI 全栈执行 | 流式产物渲染、Action 协议、多面板交互、增量解析器 |
| mastra-ai/mastra | TypeScript Agent 框架 | 工作流编排引擎、四层 Memory、评测闭环、持久化 Agent |
| OpenBMB/ChatDev | 多 Agent 协作开发 | 配置驱动编排、领域模板化、子图组合、节点类型注册表 |

## 4. Phase 1：Web 任务体验补强

### 4.1 Seahorse Web Agent Event Protocol 增强

**来源**：借鉴 CopilotKit AG-UI 的事件分族思想，但使用 Seahorse 自有命名和强类型实现，不承诺外部协议兼容。

**当前状态**：Seahorse 已有 22 种 StreamEventType（含 RUN_STARTED、STEP_STARTED、TOOL_CALL_WAITING_USER、SOURCE_FOUND、ARTIFACT_CREATED 等），前端 chatStreamUtils 已能解析这些事件。**但与 CopilotKit AG-UI 协议相比，仍缺少**：
- 事件序号（lastEventSeq）断线恢复机制
- STATE_SNAPSHOT / STATE_DELTA 状态同步事件
- 事件信封的统一结构（当前各事件 payload 格式不一致）
- 前端按事件族分发的统一 Hook API

**改进方案**：

#### 事件分类体系

```
┌─────────────────────────────────────────────────────────────┐
│              Seahorse Web Agent Event Protocol                │
├─────────────────────────────────────────────────────────────┤
│ 生命周期族                                                    │
│   RUN_STARTED / RUN_FINISHED / RUN_ERROR                    │
│   STEP_STARTED / STEP_FINISHED                              │
│                                                             │
│ 消息族                                                       │
│   TEXT_MESSAGE_START / TEXT_MESSAGE_CONTENT / TEXT_MESSAGE_END│
│   THINKING_START / THINKING_CONTENT / THINKING_END          │
│                                                             │
│ 工具族                                                       │
│   TOOL_CALL_START / TOOL_CALL_ARGS / TOOL_CALL_END          │
│   TOOL_CALL_RESULT / TOOL_CALL_WAITING_USER                 │
│                                                             │
│ 产物族                                                       │
│   ARTIFACT_START / ARTIFACT_CONTENT / ARTIFACT_END          │
│   SOURCE_FOUND / SOURCE_CARD                                │
│                                                             │
│ 状态族                                                       │
│   STATE_SNAPSHOT / STATE_DELTA                              │
│   PROGRESS_UPDATE / RECOVERABLE_ERROR                       │
└─────────────────────────────────────────────────────────────┘
```

#### 后端实现（Spring Boot）

> 注意：Kernel 侧使用强类型 domain event（现有 `StreamEventType` enum + typed payload record），Web adapter 层做 JSON 信封序列化。不使用裸 `String type` + `JsonNode payload` 作为领域主模型。

```java
// Web adapter 层的 JSON 信封（仅用于 SSE 序列化）
public record StreamEventEnvelope(
    long eventSeq,       // 递增序号，支持断线恢复
    StreamEventType type,// 强类型枚举（已有 22 种）
    String runId,
    String stepId,       // nullable
    Instant timestamp,
    Object payload       // typed payload record，序列化为 JSON
) {}

// 事件发射增强（在现有 AgentStreamTimelineEvents 基础上）
@Component
public class StreamEventSequencer {
    private final AtomicLong seq = new AtomicLong();
    
    public StreamEventEnvelope wrap(StreamEventType type, String runId, 
                                     String stepId, Object payload) {
        return new StreamEventEnvelope(
            seq.incrementAndGet(), type, runId, stepId, Instant.now(), payload);
    }
}
```

#### 前端消费（增强现有 chatStreamUtils）

```typescript
// useStreamEventReplay.ts — 在现有 useStreamResponse 基础上增加 replay 能力
function useStreamEventReplay(sessionId: string) {
  const lastSeqRef = useRef<number>(0);
  
  const connect = useCallback(() => {
    const url = `/rag/v3/chat?sessionId=${sessionId}&lastEventSeq=${lastSeqRef.current}`;
    const source = new EventSource(url);
    source.onmessage = (e) => {
      const envelope = JSON.parse(e.data);
      lastSeqRef.current = envelope.eventSeq;
      // 分发到现有 chatStreamUtils 的事件处理逻辑
      dispatchStreamEvent(envelope);
    };
    return source;
  }, [sessionId]);
  
  return { connect, lastSeq: lastSeqRef };
}
```

#### 断线恢复协议

```
客户端重连时携带 lastEventSeq:
  GET /rag/v3/chat?sessionId=xxx&lastEventSeq=42

服务端行为:
  1. 查询事件缓冲（Redis/内存，TTL 5min）
  2. 若 seq 42 之后有缓存事件 → 返回 missed events
  3. 若缓冲已过期 → 返回 STATE_SNAPSHOT（当前完整状态）
  4. 继续正常流式推送
```

**验收标准**：
- 所有 Agent 执行阶段在前端有对应事件和 UI 反馈
- 浏览器刷新后 5 秒内恢复当前任务状态
- 事件协议向后兼容，旧客户端忽略未知事件类型

### 4.2 流式产物渲染 + Action 协议

**来源**：bolt.new 的 StreamingMessageParser + Action 系统。

**当前状态**：Seahorse 已有 `AgentArtifact` 一等模型（13 字段）和前端 `AgentTracePanel`。**但与 bolt.new 相比，仍缺少**：
- 产物的流式增量渲染（当前产物在消息完成后才展示，非边生成边渲染）
- 独立的产物面板（当前产物内联在消息中）
- 结构化 Action 协议（当前事件 payload 缺少统一的 Action 类型体系）
- SwitchableStream 超长产物续接机制

**改进方案**：

#### Action 类型定义

### 4.3 内置 Action Renderer（受控富组件）

**来源**：借鉴 CopilotKit 的 `useCopilotAction` + render 模式，但当前阶段只做内置 renderer，不开放动态注册。

**当前状态**：Seahorse 前端已有 `ApprovalCard`、`SourceList`、`AgentTracePanel` 等富组件。当前阶段目标是将这些组件统一为 action renderer 模式，每个 action type 对应一个 canonical domain owner。

**改进方案**：

#### Action Type → Domain Owner 映射（安全边界）

| Action Type | Canonical Owner | 安全约束 |
|-------------|----------------|---------|
| `approval` | `ApprovalRequest` | 必须走审批领域模型，不能成为前端独立状态源 |
| `artifact` | `AgentArtifact` | 下载必须走 `AgentArtifactDownloadDecision`，不暴露 storageRef |
| `source` | `AgentRunSnapshotSource` / ContextPack | 来源可信度标注 |
| `memory_notice` | UserMemory / MemoryPrivacy | 受隐私模式约束，用户可见可删 |
| `quota_warning` | QuotaSummary / CostUsage | 只展示预估，不暴露内部 token 细节 |
| `navigate` | 前端建议（非强制） | 只能作为 UI 建议，不由模型直接触发跳转 |

## 5. Phase 2：Research Web Agent + 评测闭环

### 5.1 DurableTaskQueuePort + ResearchRunOrchestrator

**来源**：Mastra 的持久化 Agent + Deer Flow 的研究步骤清晰度 + ChatDev 的模板参数化。

**当前状态**：Seahorse 有 AgentDefinition/AgentVersion 模型和 `KernelAgentFactoryService`，但缺少声明式工作流定义。当前 Agent 执行是单一 Loop 模式（`KernelAgentLoop`），不支持多步骤编排、并行扇出、条件分支。

**关键决策**：不引入 Temporal 或任何外部 workflow engine。通过 `DurableTaskQueuePort` 抽象实现持久化任务队列，初始 adapter 用 JDBC 表，Kernel 只依赖 port 不依赖具体库。研究步骤通过 `ResearchRunOrchestrator` 固定编排，不引入任意控制流 DSL。

> **Temporal 准入条件（Phase 4+ ADR）**：只有当 bounded orchestrator 已无法满足多天长任务、跨服务补偿、复杂 fan-out/fan-in、强 worker 崩溃恢复、Workflow 历史审计时，才评估引入。当前 C 端 Web Research Agent 不满足这个门槛。

**改进方案**：

#### DurableTaskQueuePort（Kernel 端口）

```java
// DurableTaskQueuePort.java — Kernel 只依赖此接口
public interface DurableTaskQueuePort {
    
    void enqueue(String runId);
    
    Optional<DurableTask> claimNext(String workerId);
    
    void ack(String taskId);
    
    void retry(String taskId, Instant retryAt, String reason);
    
    void fail(String taskId, String reason);
    
    void cancel(String runId);
}

public record DurableTask(
    String taskId,
    String runId,
    String stepType,       // PLAN, SEARCH, FETCH, ANALYZE, WRITE, VERIFY
    int attemptCount,
    Instant createdAt,
    Instant claimedAt,
    JsonNode payload
) {}
```

#### JDBC Adapter（初始实现）

```java
// JdbcDurableTaskQueueAdapter.java
@Component
public class JdbcDurableTaskQueueAdapter implements DurableTaskQueuePort {
    private final JdbcTemplate jdbc;
    
    @Override
    public void enqueue(String runId) {
        jdbc.update("""
            INSERT INTO durable_task_queue (task_id, run_id, status, created_at)
            VALUES (?, ?, 'PENDING', NOW())
            """, UUID.randomUUID().toString(), runId);
    }
    
    @Override
    public Optional<DurableTask> claimNext(String workerId) {
        // SELECT ... WHERE status = 'PENDING' ORDER BY created_at
        // FOR UPDATE SKIP LOCKED → 避免竞争
        // UPDATE status = 'CLAIMED', worker_id = ?, claimed_at = NOW()
        return jdbc.query(CLAIM_SQL, workerId).stream().findFirst();
    }
    
    @Override
    public void ack(String taskId) {
        jdbc.update("UPDATE durable_task_queue SET status = 'DONE' WHERE task_id = ?", taskId);
    }
    
    @Override
    public void retry(String taskId, Instant retryAt, String reason) {
        jdbc.update("""
            UPDATE durable_task_queue 
            SET status = 'PENDING', retry_at = ?, attempt_count = attempt_count + 1, 
                last_error = ?
            WHERE task_id = ?
            """, retryAt, reason, taskId);
    }
    
    @Override
    public void fail(String taskId, String reason) {
        jdbc.update("""
            UPDATE durable_task_queue SET status = 'FAILED', last_error = ? WHERE task_id = ?
            """, reason, taskId);
    }
    
    @Override
    public void cancel(String runId) {
        jdbc.update("""
            UPDATE durable_task_queue SET status = 'CANCELLED' 
            WHERE run_id = ? AND status IN ('PENDING', 'CLAIMED')
            """, runId);
    }
}
```

// ResearchRunOrchestrator.java — 不是通用 workflow engine
@Service
public class ResearchRunOrchestrator {
    private final DurableTaskQueuePort taskQueue;
    private final WebSearchPort webSearch;
    private final WebFetchPort webFetch;
    
    public void startResearch(String runId, ResearchRequest request) {
        // 将研究任务拆解为固定步骤，依次入队
        taskQueue.enqueue(runId);  // 第一步：PLAN
    }
    
    // Worker 消费任务，按 stepType 分发
    @Scheduled(fixedDelay = 1000)
    public void poll() {
        taskQueue.claimNext(workerId).ifPresent(task -> {
            try {
                var result = executeStep(task);
                taskQueue.ack(task.taskId());
                enqueueNextStep(task, result);  // 完成后入队下一步
            } catch (RetryableException e) {
                taskQueue.retry(task.taskId(), 
                    Instant.now().plusSeconds(30), e.getMessage());
            } catch (Exception e) {
                taskQueue.fail(task.taskId(), e.getMessage());
            }
        });
    }
    
    private Object executeStep(DurableTask task) {
        return switch (ResearchStepType.valueOf(task.stepType())) {
            case PLAN -> planStep(task);
            case SEARCH -> searchStep(task);
            case FETCH -> fetchStep(task);
            case ANALYZE -> analyzeStep(task);
            case WRITE -> writeStep(task);
            case VERIFY -> verifyStep(task);
        };
    }
}

enum ResearchStepType { PLAN, SEARCH, FETCH, ANALYZE, WRITE, VERIFY }

// ObservationMemoryExtractor.java — 异步防抖提取
@Service
public class ObservationMemoryExtractor {
    private final Map<String, ScheduledFuture<?>> pendingExtractions = new ConcurrentHashMap<>();
    private static final Duration DEBOUNCE = Duration.ofSeconds(30);
    
    public void scheduleExtraction(String userId, String threadId, List<Message> newMessages) {
        String key = userId + ":" + threadId;
        
        // 取消之前的待执行任务（防抖）
        ScheduledFuture<?> existing = pendingExtractions.get(key);
        if (existing != null) existing.cancel(false);
        
        // 重新调度
        ScheduledFuture<?> future = scheduler.schedule(
            () -> extractObservations(userId, newMessages),
            DEBOUNCE
        );
        pendingExtractions.put(key, future);
    }
    
    private void extractObservations(String userId, List<Message> messages) {
        // 调用 LLM 提取事实、偏好、纠正信号
        var facts = llmExtractor.extract(messages);
        // 去重合并
        observationMemoryPort.mergeObservations(userId, facts);
    }
}

// EvalPipeline.java — 五阶段管道
@Service
public class EvalPipeline {
    
    public EvalResult evaluate(EvalSample sample, List<Evaluator> evaluators) {
        var scores = new ArrayList<EvalScore>();
        
        for (Evaluator evaluator : evaluators) {
            // 1. preprocess: 标准化输入
            var prepared = evaluator.preprocess(sample);
            // 2. extract: 提取评测特征
            var features = evaluator.extract(prepared);
            // 3. analyze: 分析特征
            var analysis = evaluator.analyze(features);
            // 4. generateScore: 生成分数
            var score = evaluator.score(analysis);
            // 5. emit: 发射结果
            scores.add(score);
        }
        
        return new EvalResult(sample.getId(), scores);
    }
}

### 7.2 中间件驱动的上下文工程

**来源**：Deer Flow 的 13+ 中间件链 + ID 交换前缀缓存优化。

**当前状态**：Seahorse 的 `ContextPackRuntimeAssembler` 和 `ConversationAttachmentContextAssembler` 提供了上下文组装能力，但逻辑集中在少数服务中，缺少可组合的中间件管道和前缀缓存优化。

**改进方案**：

#### 中间件链设计

```java
// AgentMiddleware.java
public interface AgentMiddleware {
    int order();  // 执行顺序
    AgentContext process(AgentContext ctx);
}

// 中间件注册（按 order 排序执行）
@Configuration
public class MiddlewareChainConfig {
    @Bean
    public List<AgentMiddleware> middlewareChain() {
        return List.of(
            new MessageHistoryMiddleware(),       // 1. 消息历史截断
            new WorkingMemoryMiddleware(),        // 2. 工作记忆注入
            new SemanticRecallMiddleware(),       // 3. 语义检索
            new ObservationMemoryMiddleware(),    // 4. 观察记忆
            new SkillTriggerMiddleware(),         // 5. 技能触发和 L2 加载
            new ToolPolicyMiddleware(),           // 6. 工具策略过滤
            new DynamicContextMiddleware(),       // 7. 动态上下文（日期、用户画像）
            new TokenBudgetMiddleware(),          // 8. Token 预算裁剪
            new SummarizationMiddleware(),        // 9. 超长上下文摘要
            new PrefixCacheOptimizer()            // 10. 前缀缓存位置优化
        );
    }
}

// PrefixCacheOptimizer.java — ID 交换技术（借鉴 Deer Flow）
@Component
public class PrefixCacheOptimizer implements AgentMiddleware {
    
    @Override
    public AgentContext process(AgentContext ctx) {
        // 将系统提示词中的动态内容（记忆、日期等）
        // 通过 stable ID 标记位置，确保前缀部分不变
        // 从而最大化 LLM 的前缀缓存命中率
        var systemMessages = ctx.getSystemMessages();
        for (var msg : systemMessages) {
            if (msg.hasDynamicContent()) {
                msg.setId(stableIdFor(msg.getRole()));  // 固定 ID
            }
        }
        return ctx;
    }
}
```

**验收标准**：
- 中间件可独立测试，新增中间件不影响已有逻辑
- 前缀缓存命中率可观测（通过 tracing 记录）
- Token 预算裁剪确保不超出模型上下文窗口

### 7.3 双向状态同步

**来源**：CopilotKit 的 `useCopilotReadable` + `STATE_SNAPSHOT` 事件。

**当前状态**：Seahorse 前端 `chatStreamUtils` 已能解析多种事件类型并更新 store，但前端应用状态对 Agent 不可见（Agent 不知道用户当前在看什么页面、时区、偏好），Agent 也无法主动推送状态变更到前端。

**改进方案**：

#### 前端状态注册

```typescript
// useAgentReadable.ts — 让 Agent 感知前端状态
function useAgentReadable(config: {
  key: string;
  description: string;
  value: any;
}) {
  const stateRegistry = useStateRegistry();
  
  useEffect(() => {
    stateRegistry.register(config.key, {
      description: config.description,
      value: config.value
    });
    return () => stateRegistry.unregister(config.key);
  }, [config.value]);
}

// 使用示例
function ChatPage() {
  const { currentSession } = useSessionStore();
  
  useAgentReadable({
    key: 'current_session',
    description: '用户当前会话信息',
    value: { 
      sessionId: currentSession.id, 
      messageCount: currentSession.messages.length,
      topic: currentSession.title 
    }
  });
  
  useAgentReadable({
    key: 'user_timezone',
    description: '用户时区',
    value: Intl.DateTimeFormat().resolvedOptions().timeZone
  });
}
```

#### 后端状态消费

### 7.4 外部 Workflow Engine（Temporal 等，需 ADR）

**来源**：Mastra 的三种持久化模式 + Deer Flow 的 Checkpoint。

**当前状态**：Phase 2 通过 `DurableTaskQueuePort`（JDBC 实现）解决持久化执行。当 bounded orchestrator 遇到以下瓶颈时，才评估引入 Temporal：
- 多天长任务（当前研究任务 < 10 分钟）
- 跨服务补偿事务
- 复杂 fan-out/fan-in（当前只有线性步骤）
- 强 worker 崩溃恢复（当前 JDBC `FOR UPDATE SKIP LOCKED` 足够）
- Workflow 历史审计（当前 AgentRun/AgentStep 已覆盖）

**升级路径**：JDBC → db-scheduler/JobRunr → Temporal（每步需 ADR）

**改进方案**：

#### 挂起/恢复协议

```java
// AgentSuspendService.java
@Service
public class AgentSuspendService {
    
    // 工具调用需要用户确认时挂起
    public SuspendResult suspendForUserConfirm(String runId, String stepId, 
                                                ToolCallRequest toolCall) {
        // 1. 序列化当前状态
        var checkpoint = AgentCheckpoint.builder()
            .runId(runId)
            .stepId(stepId)
            .stateJson(serializeState(runId))
            .messageHistoryJson(serializeMessages(runId))
            .pendingToolCallJson(objectMapper.writeValueAsString(toolCall))
            .reason(SuspendReason.USER_CONFIRM)
            .build();
        
        // 2. 持久化
        checkpointRepository.save(checkpoint);
        
        // 3. 发送 SSE 事件通知前端
        eventEmitter.emit(AgUiEvent.of("TOOL_CALL_WAITING_USER", runId, stepId,
            Map.of("toolCallId", toolCall.getId(), 
                   "toolName", toolCall.getToolName(),
                   "reason", toolCall.getRiskReason())));
        
        return new SuspendResult(checkpoint.getId());
    }
    
    // 用户确认后恢复
    public Flux<AgUiEvent> resume(String checkpointId, UserDecision decision) {
        var checkpoint = checkpointRepository.findById(checkpointId);
        
        // 恢复状态
        var state = deserializeState(checkpoint.getStateJson());
        var messages = deserializeMessages(checkpoint.getMessageHistoryJson());
        var pendingToolCall = deserializeToolCall(checkpoint.getPendingToolCallJson());
        
        // 根据用户决策继续或跳过
        if (decision.isApproved()) {
            return agentLoop.resumeWithToolResult(state, messages, pendingToolCall);
        } else {
            return agentLoop.resumeWithToolSkipped(state, messages, pendingToolCall, 
                                                    decision.getReason());
        }
    }
}
```

**验收标准**：
- 高风险工具调用自动挂起等待用户确认
- 用户确认后 Agent 从断点恢复，不重新执行已完成步骤
- 浏览器关闭后重新打开，挂起的任务仍可恢复

### 7.5 MCP 会话池化（企业扩展，默认关闭）

**来源**：Deer Flow 的 MCPSessionPool + LRU 驱逐。

**当前状态**：Seahorse 有 `seahorse-agent-adapter-mcp-http`（StreamableHttpMcpClient），但每次工具调用可能创建新连接，有状态工具（如 Playwright）无法保持会话连续性。

**改进方案**：

```java
// McpSessionPool.java
@Component
public class McpSessionPool {
    private final int MAX_SESSIONS = 256;
    private final LinkedHashMap<SessionKey, McpSession> pool = 
        new LinkedHashMap<>(MAX_SESSIONS, 0.75f, true) {  // accessOrder=true → LRU
            @Override
            protected boolean removeEldestEntry(Map.Entry<SessionKey, McpSession> eldest) {
                if (size() > MAX_SESSIONS) {
                    eldest.getValue().close();
                    return true;
                }
                return false;
            }
        };
    
    public synchronized McpSession getOrCreate(String serverName, String scopeKey) {
        var key = new SessionKey(serverName, scopeKey);
        return pool.computeIfAbsent(key, k -> createSession(serverName));
    }
    
    // scopeKey 通常是 userId + threadId，确保同一用户同一会话复用连接
    record SessionKey(String serverName, String scopeKey) {}
}
```

**验收标准**：
- 同一用户同一会话内的 MCP 工具调用复用连接
- 会话池有 LRU 驱逐，不会无限增长
- 有状态工具（如浏览器自动化）跨多次调用保持状态

## 10. 总结

本方案从 5 个开源项目中提炼了 14 项可借鉴的技术亮点，经过 C 端 Web 适配筛选后，按 Phase 文档对齐为四个阶段：

- **Phase 1（体验补强）**：事件信封版本化、lastEventSeq replay、独立 Artifact 面板、内置 Action Renderer — 补齐协议完整性和 UX 体验
- **Phase 2（Research Agent）**：DurableTaskQueuePort + ResearchRunOrchestrator、Web 搜索/抓取、渐进式技能加载、评测最小闭环 — 实现 bounded research pipeline
- **Phase 3（个性化运营）**：四层 Memory 深化、领域模板库、中间件上下文工程 — 实现用户留存和质量治理
- **Phase 4+（高级扩展）**：步骤级并行、MCP 池化、双向状态同步、生成式 UI 注册表 — 按需引入，每项需 ADR

**核心原则**：
1. **吸收产品体验和工程原则，而非照搬开源框架形态**
2. **DurableTaskQueuePort 解决持久化执行**，JDBC 初始实现，后续按需升级 adapter（db-scheduler → JobRunr → Temporal）
3. **不引入外部 workflow engine**，Temporal 准入条件为 Phase 4+ ADR（多天长任务、跨服务补偿、强 worker 崩溃恢复）
3. **Action type 必须有 canonical domain owner**，不允许绕过现有领域模型
4. **不创建 child agent identity**，步骤级并行通过 DurableTaskQueue 多任务入队实现
5. **模板配置参数 + 绑定预建 Orchestrator**，不允许用户表达任意控制流
