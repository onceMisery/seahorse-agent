2.3.2 重试策略增强：

改造 `RetryableResearchException`：
```java
public class RetryableResearchException extends RuntimeException {
    private final boolean retryable;
    private final Set<Class<? extends Exception>> nonRetryableTypes = Set.of(
        SecurityException.class,        // 安全拒绝不重试
        IllegalArgumentException.class  // 参数错误不重试
    );

    public boolean shouldRetry(int attemptCount, int maxAttempts) {
        if (!retryable) return false;
        if (attemptCount >= maxAttempts) return false;
        // 遍历异常链，命中黑名单则不重试
        Throwable cause = getCause();
        while (cause != null) {
            if (nonRetryableTypes.contains(cause.getClass())) return false;
            cause = cause.getCause();
        }
        return true;
    }
}
```

**关键文件：**
- `seahorse-agent-kernel/.../application/agent/research/ResearchLoopDetector.java`（新增）
- `seahorse-agent-kernel/.../application/agent/research/ResearchRunOrchestrator.java`
- `seahorse-agent-kernel/.../application/agent/research/RetryableResearchException.java`

**验收标准：**

- 安全类异常不重试，直接 fail

### 2.4 WebSource Trust Level 评估（P1）— 吸收 ChatDev 混合重排序

**来源亮点：**
- ChatDev 的混合重排序：70% embedding 相似度 + 30% 多信号语义（Jaccard + LCS + keyword + length）
- DeerFlow 的 memory confidence scoring：结构化事实 + 置信度分数

**落地设计：**

新增 `SourceTrustEvaluator`：
```java
public class SourceTrustEvaluator {

    public SourceTrustLevel evaluate(WebSource source) {
        double score = 0.0;

        // 信号 1：HTTPS（权重 0.2）
        if (source.url().startsWith("https://")) score += 0.2;

        // 信号 2：域名信誉（权重 0.4）— 基于已知高信誉域名列表
        score += domainReputationScore(source.url()) * 0.4;

        // 信号 3：抓取新鲜度（权重 0.2）— 24h 内满分，7 天后 0 分
        score += freshnessScore(source.retrievedAt()) * 0.2;

        // 信号 4：内容长度合理性（权重 0.2）— 太短或太长扣分
        score += contentLengthScore(source.contentHash()) * 0.2;

        if (score >= 0.7) return SourceTrustLevel.HIGH;
        if (score >= 0.4) return SourceTrustLevel.MEDIUM;
        if (score >= 0.2) return SourceTrustLevel.LOW;
        return SourceTrustLevel.UNTRUSTED;
    }
}
```

域名信誉列表（初始版本，可配置化）：
- HIGH (0.9-1.0)：wikipedia.org, nature.com, science.org, gov.cn, edu.cn
- MEDIUM (0.5-0.7)：主流新闻、技术博客
- LOW (0.2-0.4)：未知域名但有 HTTPS
- UNTRUSTED (0.0)：HTTP、已知垃圾站

集成到 `SearchStepHandler`：
- 搜索结果返回后立即评估 trust level
- 写入 `WebSource.trustLevel`
- FETCH 阶段优先抓取 HIGH/MEDIUM 来源

前端 `SourceList.tsx`：
- 展示 trust level 标签（高可信 / 中等 / 低可信）
- 颜色区分：绿/黄/灰

**关键文件：**
- `seahorse-agent-kernel/.../application/agent/research/SourceTrustEvaluator.java`（新增）
- `seahorse-agent-kernel/.../application/agent/research/SearchStepHandler.java`
- `frontend/src/components/chat/SourceList.tsx`

### 2.6 Eval 评估维度（P1）— 吸收 Mastra 评估器生态

**来源亮点：**
- Mastra 的 11+ LLM 评估器：Faithfulness、Hallucination、Bias、Toxicity 等
- Mastra 的五阶段评估管线：preprocess → extract → analyze → score → reason
- Mastra 的 cost tracking：自动定价注册表 + token 用量分解

**落地设计：**

为 `KernelEvalRegressionService` 定义评估维度：

```java
public enum EvalDimension {
    ANSWER_QUALITY,        // 回答是否准确、完整、有帮助
    SOURCE_GROUNDEDNESS,   // 结论是否有来源支撑
    CITATION_COMPLETENESS, // 引用是否完整、可追溯
    TASK_COMPLETION,       // 任务是否完成（研究报告是否生成）
    LATENCY,              // 端到端耗时
    COST_PER_TASK         // 单任务成本
}
```

评估流程（五阶段）：
1. **Preprocess**：从 eval sample 中提取 question、expected_output、actual_output、sources、citations
2. **Extract**：提取关键事实、引用关系、来源映射
3. **Analyze**：LLM 判断 groundedness（结论是否有来源支撑）、completeness（是否遗漏关键信息）
4. **Score**：每个维度 0-1 分
5. **Reason**：生成人可读的评估理由

初始版本只实现 `CITATION_COMPLETENESS`（复用 `CitationVerifier`）和 `TASK_COMPLETION`（检查是否生成 Artifact）两个自动化维度，其余维度标记为「需人工评估」。

**关键文件：**
- `seahorse-agent-kernel/.../application/agent/eval/EvalDimension.java`（新增）
- `seahorse-agent-kernel/.../application/agent/eval/KernelEvalRegressionService.java`

### 2.7 ContextPack 前缀缓存优化（P2）— 吸收 DeerFlow 冻结快照

**来源亮点：**
- DeerFlow 的 Frozen Snapshot Mode：系统 prompt 完全静态，动态内容注入为 `<system-reminder>` HumanMessage
- 目的：最大化 LLM prefix cache 命中率（跨用户/会话复用）

**落地设计：**

改造 `ContextPackRuntimeAssembler`：
- 系统 prompt 模板保持静态（不含用户名、时间、记忆等动态内容）
- 动态内容（当前时间、用户记忆、附件摘要）作为独立 context item 注入，放在 system prompt 之后
- 确保 system prompt 的 token 序列在不同用户/会话间完全一致

效果：OpenAI/Anthropic 的 prefix cache 能命中 system prompt 部分，减少首 token 延迟和成本。

> 注意：这是 P2 优化，不影响功能正确性，可在 P0/P1 完成后再做。

### 3.2 Approval 防闪烁状态机（P1）— 吸收 CopilotKit 两阶段 HITL

- CopilotKit `useInterrupt` 的两阶段状态机：

**落地设计：**

改造 `chatStore.ts` 中 approval 处理逻辑：

```typescript
// 累积阶段：收到 tool_call_waiting_user 时暂存
let pendingApprovals: AgentApproval[] = [];

// 在 step_finished 或 run 暂停时才 promote
function promotePendingApprovals() {
  if (pendingApprovals.length > 0) {
    set({ approvals: [...get().approvals, ...pendingApprovals] });
```

触发 promote 的时机：
- 收到 `step_finished` 且 step status 为 `WAITING_USER`

### 3.3 乐观更新模式（P1）— 吸收 bolt.new Nanostores 乐观写入

**来源亮点：**
- bolt.new：文件修改立即更新 UI，异步 flush 到 WebContainer
- bolt.new：HMR 状态保持——`import.meta.hot?.data` 保存 store 实例，热更新不丢状态

**落地设计：**

适用场景：
1. **附件上传**：选择文件后立即在 ChatInput 显示缩略图 + "上传中" 状态，API 返回后更新为真实 attachmentId
2. **反馈提交**：点踩后立即切换按钮状态为 "已反馈"，API 失败时回滚
3. **Approval 决策**：点击 approve 后立即隐藏卡片 + 显示 "已批准" 标签，API 失败时恢复卡片

实现模式：
```typescript
// 通用乐观更新 helper
function optimisticUpdate<T>(
  getCurrentState: () => T,
  applyOptimistic: (state: T) => T,
  apiCall: () => Promise<void>,
  rollback: (state: T) => void
) {
  const prev = getCurrentState();
  applyOptimistic(prev);
  apiCall().catch(() => rollback(prev));
}
```

**验收标准：**
- 附件上传：选择文件后 <100ms 内显示缩略图
- 反馈：点击后 <50ms 内按钮状态变化
- API 失败时 UI 回滚到之前状态 + toast 提示

### 3.4 细粒度订阅与派生 Store（P1）— 吸收 bolt.new Computed Store + CopilotKit 版本计数器

```typescript
export const useArtifactStore = create<ArtifactSlice>((set) => ({
```

### 3.7 错误安全订阅守卫（P1）— 吸收 CopilotKit 错误隔离

**来源亮点：**
- CopilotKit：所有 subscriber callback 包裹在 error-safe guard 中，捕获同步 throw 和异步 rejection
- 单个 subscriber 失败不中断通知循环

**落地设计：**

`useStreamResponse.ts` 中 handler 调用改造：
```typescript
function safeInvoke<Args extends unknown[]>(
  handler: ((...args: Args) => unknown) | undefined,
  ...args: Args
) {
  if (!handler) return;
  try {
    const result = handler(...args);
    if (result && typeof (result as PromiseLike<unknown>).then === "function") {
      Promise.resolve(result).catch((error) => {
        console.error("[stream] async handler error, skipping:", error);
      });
    }
  } catch (error) {
    console.error("[stream] handler error, skipping:", error);
  }
}
```

`useStreamResponse.ts` 中 SSE 续传处理：
```typescript
// readSseStream 未收到 done/finish/cancel/error 就断开时抛错
// streamWithRetry 后续请求只使用 resumeRunId/lastEventSeq，不重复提交原始问题
const checkpoint = shouldResumeNextAttempt ? resume?.() : null;
const nextUrl = checkpoint?.runId
  ? resumeUrl(url, checkpoint.runId, checkpoint.lastEventSeq)
  : url;
```

**验收标准：**
- 单个事件处理器同步抛异常或异步 rejection 不影响流读取和其他处理器
- `stream_event` envelope 与同名 typed SSE 事件去重，避免重复消费
- SSE 非终态断开或 watchdog 超时后，最多 3 次指数退避重试
- 已拿到 `runId/lastEventSeq` 时通过恢复 URL 续传；未拿到断点时不重发原始问题，避免重复创建长任务
- 后端 `SeahorseChatControllerReplayTests` 覆盖 replay、snapshot fallback、replay 后继续实时流、snapshot 可续传后继续实时流
