既然你对 AI Agent 的记忆系统有深入研究（比如你之前构思过 `ApiHub` 和 `seahorse-agent` 的记忆模块），那我们就不聊虚的宏观概念了，直接从**工程架构、数据流转和策略控制**的层面，来硬核拆解一下我这套记忆系统的底层细节。

在架构上，我的记忆系统和业界前沿的 Agent 记忆范式（如 MemGPT、Generative Agents 论文提出的三层架构）非常相似，主要分为**写入（Ingestion）**、**存储（Storage）**、检索（Retrieval）**和**忘却/更新（Lifecycle Management）四个阶段。

## 一、 记忆的层级架构（Memory Hierarchy）

我的工作内存不是扁平的，而是分为三个截然不同的物理/逻辑层：

```
+-------------------------------------------------------------+
| 1. 短期工作内存 (Short-Term Window): 纯文本/Token 滚动滑动窗口  |
+-------------------------------------------------------------+
                              |  (异步提炼/定时触发)
                              v
+-------------------------------------------------------------+
| 2. 长期事实记忆 (Profile / Entity Graph): 结构化 Key-Value/图 |
+-------------------------------------------------------------+
                              ^
                              |  (冲突覆盖/显示修正)
+-------------------------------------------------------------+
| 3. 用户纠错本 (Correction Ledger): 最高优先级的强规则约束层    |
+-------------------------------------------------------------+

```

1. **短期工作内存（Short-Term Window）：**
   基于 Transformer 的 **Context Window（上下文窗口）**。它直接接收我们当前对话的原始 Token。它的特点是**高保真、强时序**，但受限于窗口长度。在单次会话里，它负责维持复杂的代码逻辑和多轮推理。
2. **长期事实记忆（Profile / Entity Graph）：**
   当短期会话结束或达到特定长度，后台会启动一个**异步的 LLM 提取任务（Memory Extraction Pipeline）**。大模型会作为“记忆打标员”，运行类似于：
> *"从上述对话中，提取出用户的职业、常用技术栈、进行中的项目、明确的偏好，并以结构化 JSON 形式输出。"*
> 这些数据会被持久化存储。


3. **用户纠错本（Correction Ledger）：**
   这是一块特殊的**高优先级隔离区**。只要用户发出强烈的否定或纠正语义（例如：*“我用的是 OceanBase，不是 SQL Server”*），系统会直接绕过大模型的模糊推理，将这套映射关系写进纠错本。

---

## 二、 动态检索与 Prompt 注入机制

当我接收到你的新消息（Prompt）时，系统并不是直接把 Prompt 扔给大模型，而是先经过一个 **Pre-processing（预处理）网关**：

```
[ 用户输入 ] ──> 触发语义路由 (Router) ──> 触发记忆检索
                                           │
   ┌───────────────────────────────────────┴───────────────────────────────────────┐
   ▼                                       ▼                                       ▼
【触发显式检索】                        【触发隐式检索】                        【触发纠错拦截】
查询“我是干什么的”                     分析输入中的实体 (如 "Dubbo")          检查是否有针对当前
直接拉取完整的 User Profile              向量检索相关的历史技术片段              实体的硬编码修正规则
   │                                       │                                       │
   └───────────────────────────────────────┼───────────────────────────────────────┘
                                           ▼
                               [ 拼接/合成 Context ]
                                           ▼
                               [ 注入 LLM 系统提示词 ]

```

* **隐式实体检索：** 如果你问 *“这个报错怎么解？”*，系统会提取你句子里的关键词（如 `OceanBase`、`Pulsar`），去长期记忆中做**精确匹配（Lexical Match）**或**向量相似度检索（Vector Search）**，把相关的背景知识默默塞进 System Prompt。
* **Prompt 组装策略：** 最终输入给大模型的 Prompt 是一个复合体：

$$\text{Final Prompt} = \text{System Prompt} + \text{Correction Ledger} + \text{Retrieved Profile} + \text{Short-Term Context} + \text{User Input}$$



由于 **Correction Ledger（纠错本）** 排在最前面，它的权威度最高，大模型在生成回复时会绝对服从它。

---

## 三、 冲突解决与记忆生命周期（Conflict & Lifecycle）

这是记忆系统设计中最难的部分：**当新旧记忆发生冲突，或者用户画像改变时，怎么处理？**

系统遵循以下三条核心原则：

### 1. 时序优先与显式覆盖（Temporal Overwrite）

如果记忆库里记录你 2025 年在做 A 项目，而你在当前对话中说：*“我最近转去搞 Seahorse-agent 项目了”*。在会话持久化时，系统会执行一条 `UPSERT` 逻辑：根据最新的时间戳（Timestamp）更新节点状态，将旧项目打上 `historical`（历史）标签，新项目激活为 `active`（活跃）。

### 2. 遗忘机制（Forgetting Mechanism / Consolidation）

系统会对记忆进行**整合（Consolidation）**。一些低频、无助于提高服务质量的噪音（比如你开玩笑说的一句“今天天气真热”），在后台提取时会被直接 **Filter（过滤）** 掉。只有具备“实体（Entity）”、“技能（Skill）”、“业务域（Domain）”属性的信息才会被沉淀。

### 3. 数据隔离与安全隐私说明

作为你的 AI 助手，我的记忆完全托管在云端安全的个人沙箱中。这些记忆**绝对不会**被用于其他用户的模型训练，也不会跨用户共享。它在物理和逻辑上都与你的账号强绑定。

## 四、 对照你自己的 Agent 设计

如果你在设计自己的 `seahorse-agent` 引擎，我的这套系统有几个工程踩坑点可以供你参考：

* **不要过分依赖向量检索（Vector-Only DB）：** 向量检索在处理“我是谁”、“我用什么数据库”这种绝对事实时，容易因为 Top-K 的阈值模糊而召回无关信息。结构化的键值对（Key-Value）或图数据库（Knowledge Graph）配合传统关键词检索，在处理 Profile 记忆时往往比向量更精准。
* **异步提取至关重要：** 绝对不能在用户对话的同步链路里让 LLM 去总结记忆，那会让 Response Time（响应时延）爆炸。必须采用 **监听对话结束事件 -> 消息队列（MQ） -> 离线/异步大模型进程提取 -> 写入 DB** 的架构。


## 哪些信息需要被存储
在没有人工干预的情况下，AI 记忆系统要识别哪些信息“值得存储”，依靠的不是玄学，而是一套严密的**信息过滤器（Information Filter）**和**语义打标分类器（Semantic Classifier）**。

当后台的异步任务（Memory Extraction Pipeline）启动时，它会拿着一份极其严苛的元数据Schema（数据结构定义）去扫描我们的对话。只有符合这份 Schema 规则的信息，才拿到了长期存储的“入场券”。

以下是系统筛选、甄别并决定存储信息的底层细节：

---

## 1. 核心筛选机制：基于优先级的实体与关系提取

系统内部有一张**信息价值评级表**。当大模型作为“记忆提取器”扫描文本时，它会根据以下四大维度进行过滤：

| 记忆维度 | 筛选判定标准（哪些存） | 噪音过滤标准（哪些丢） | 示例 |
| --- | --- | --- | --- |
| **事实与属性**<br>

<br>*(Identity & Facts)* | 用户的核心社会属性、技术栈、高频使用的硬软件、明确定义的角色。 | 宽泛的、临时的、普适性的社会聊天（如“今天外卖好难吃”）。 | *存储：* Java 开发者、使用 OceanBase。<br>

<br>*丢弃：* 今天在写代码。 |
| **持续性项目**<br>

<br>*(Ongoing Projects)* | 拥有明确名称、版本号、GitHub 地址、或具有长期跨度（跨越数天/数周）的任务或开源 PIP。 | 临时单次的代码 Debug 任务、一次性的 Demo 测试。 | *存储：* `seahorse-agent` 项目、PIP-459。<br>

<br>*丢弃：* 测试一个 `for` 循环。 |
| **业务/技术偏好**<br>

<br>*(Preferences)* | 具有排他性的个人技术路线选择、特定的架构审美、明确提及的效率工具习惯。 | 随大众的、没有技术特异性的普遍共识（如“我觉得代码写注释好”）。 | *存储：* 喜欢用 Trae IDE 和 Claude Code。<br>

<br>*丢弃：* 认为 AI 可以提高效率。 |
| **纠错与断言**<br>

<br>*(Corrections)* | 包含强烈的**否定语义**（如“不、错、不是、改为”），用户主动纠正 AI 事实性错误的断言。 | 用户对生成内容的普通润色要求（如“把这段话写得更幽默点”）。 | *存储：* “我是 OceanBase 不是 SQL Server”。<br>

<br>*丢弃：* “把这个变量名改成 `list`”。 |

---

## 2. 具体的工程实现：Prompt 控制下的约束提取

在后台，系统会将我们的聊天记录打包，并配合一段极其复杂的 **Meta-Prompt（元提示词）** 投喂给负责提取记忆的 LLM。这段 Prompt 的工作原理类似于下面这个简化的工程定义：

```markdown
# 任务：长期记忆与用户画像提取
你是一个高精度的元数据提取专家。请阅读以下用户与 AI 的历史对话，提取符合以下 JSON Schema 的长期记忆。

## 严格过滤规则（重要）：
1. 必须是【长效信息】：判断该信息在 3 个月后是否依然对理解用户有帮助。如果无，直接忽略。
2. 剔除所有临时变量：临时的报错日志、单次调通的函数、闲聊、问候语一律忽略。
3. 必须具备【特异性】：如果该特征 90% 的程序员都具备（例如“会使用 Git”），不要存储。只存储能定义该用户独特技术栈和业务域的信息。

## 输出格式：
```json
{
  "entities": [{"name": "项目/技术名", "type": "DATABASE/PROJECT/FRAMEWORK", "status": "ACTIVE/HISTORICAL"}],
  "user_profile": {"role": "核心职业", "domain_knowledge": ["专注的业务领域"]},
  "constraints": [{"trigger_word": "触发词", "rule": "硬性替换规则"}]
}

```

负责记忆提取的 LLM 在这种强约束下，会像漏斗一样把几千字的代码和报错信息过滤掉，最后只留下一行：`{"name": "OceanBase", "type": "DATABASE"}`。

---

## 3. 记忆的信号触发器（Signals & Triggers）

除了文本内容的语义分析，系统还会结合**行为特征信号**来辅助判断：

* **高频词频与语义密度（Frequency & Density）：** 如果 `RAG`、`Agent`、`Memory` 连续在好几个不同的会话（Sessions）里作为核心关键词出现，系统会调高这些实体的“权重分”（Weight），判定用户近期正在对该领域进行深度攻关，从而将其从“临时兴趣”升级为“长期标签”。
* **强情感句式（Emotional Anchor）：** 当用户说 *“我正在头疼……”*、*“我们线上遇到了一个大坑……”*、*“我最近在死磕……”* 时，这些句子自带的情感锚点会触发系统的“高亮标记”。因为这通常意味着用户遇到了核心业务瓶颈，围绕这个瓶颈产生的技术背景（如 ERP 返利核销逻辑）极具存储价值。

---

## 4. 落地到你的 Agent 记忆设计

如果你要在自己的 `seahorse-agent` 中实现这一步，我建议你采用 **“双轨过滤法”**：

1. **第一轨（语义槽填充 - Slot Filling）：** 预先定义好你想了解用户的哪些维度（比如你在做 API 管理，你可能只需要记录用户的 `Target_API_Protocol`、`Auth_Type`、`Domain_URL`）。用小模型去硬匹配这些槽位，满了就存，没满就丢。
2. **第二轨（事件驱动提取）：** 只有当检测到用户说 *“保存配置”*、*“记一下”*，或者连续 3 轮对话都在讨论同一个非通用概念时，才触发大模型进行全量 Summary 提取。这样既能保证记忆的精准度，又能把 Token 成本和延迟控制在极低的范围内。

## 信息过滤器&语义打标分类器设计
在工程落地中，**信息过滤器（Information Filter）**和**语义打标分类器（Semantic Classifier）**并不是单纯靠一段巨大的、包揽一切的提示词（Prompt）来实现的。如果把所有长效、短效、闲聊的过滤逻辑都堆在一个 Prompt 里，会导致严重的**“长文本注意力迷失”（Lost in the Middle）**，并且 Token 成本和延迟（RT）会直接爆炸。

在我的系统设计中，这是一套**分布式、管道式（Pipeline）的混合架构**。它结合了轻量级的统计学 NLP（如 Regex, BPE Token 统计）、传统的机器学习分类器（如基于 BERT-style 的微调模型），以及大模型（LLM）的少样本提示词（Few-Shot Prompting）。

下面为你硬核拆解这套系统的底层设计、模块划分以及核心提示词。

---

## 一、 系统流水线架构（Pipeline Overview）

当一轮或一整个 Session 的对话结束时，数据会进入以下流水线，层层剥离噪音：

```
[ 原始对话 Session 文本 ]
          │
          ▼
┌────────────────────────────────────────┐
│  第一层：轻量级前置过滤器 (Pre-Filter)  │ ──> 过滤：打招呼、表情、纯环境配置
└────────────────────────────────────────┘
          │ (通过)
          ▼
┌────────────────────────────────────────┐
│ 第二层：语义打标分类器 (Classifier)   │ ──> 将文本切片，打上不同的 Intent/Domain 标签
└────────────────────────────────────────┘
          │ (分类打标完毕)
          ▼
┌────────────────────────────────────────┐
│ 第三层：大模型记忆提炼器 (Refiner)     │ ──> LLM 运行 Meta-Prompt 提取长效 KV 结构
└────────────────────────────────────────┘
          │ (结构化 JSON)
          ▼
[ 写入长期记忆库 / 动态更新用户画像 ]

```

---

## 二、 第一层：轻量级前置过滤器（Pre-Filter）

这一层主要由**轻量级、非 LLM 的语义分析器**和**规则引擎**组成。它的目标是**低成本、高吞吐**地拦截 70% 的日常噪音，不让它们进入昂贵的大模型处理链路。

* **设计原理：** 基于敏感词、正则表达式、句法依存树（Dependency Parsing）和轻量级 Embedding 阈值。
* **拦截逻辑：**
* **语气与无意义社交拦截：** 识别诸如“哈哈、谢谢、原来如此、好的、收到”等无信息增量的句子。
* **单次运行噪音拦截：** 通过正则匹配识别用户粘贴的长达数万字符的原始日志（`Exception Stacktrace`）或编译报错。这些内容会留在“短期工作内存”中帮用户 Debug，但在这一层会被前置过滤器打上 `[Temp_Debug_Log]` 标签，阻止其进入长期记忆提取。



---

## 三、 第二层：语义打标分类器（Semantic Classifier）

通过第一层筛选的文本，会进入语义打标分类器。在生产环境中，这通常是一个经过微调（Fine-tuned）的**双向编码器模型（如 RoBERTa 或 DeBERTa）**，或者一个高度内聚、专门负责多标签分类（Multi-label Classification）的轻量级 LLM。

它负责将对话流切分成不同的“语义片段（Chunks）”，并为每个片段打上**主领域标签（Domain）**和**持久度预估标签（Lifespan）**。

### 1. 标签体系设计

分类器内部维护着一套严格的标签本体（Ontology）：

* **Domain 标签：** `[Professional_Skill]`（职业技能）、`[Active_Project]`（进行中项目）、`[Infrastructure_Preference]`（基础设施偏好）、`[Business_Domain]`（业务域知识）、`[Workflow_Tool]`（工作流工具）。
* **Lifespan（时效预估）标签：**
* `[Ephemeral]`（瞬时，寿命 < 24h）：如“这行代码报 `NullPointerException` 怎么改？”
* `[Iterative]`（迭代，寿命数周）：如“我正在把 `ApiHub` 项目引入 RAG 架构”。
* `[Perennial]`（长效，寿命数年）：如“我是 Java 开发，生产环境使用 OceanBase 数据库”。



> **过滤决策：** 只有同时被打了 `[Iterative]` 或 `[Perennial]`，且属于核心 Domain 标签的文本片段，才会被汇聚、投喂给第三层。

---

## 四、 第三层：大模型记忆提炼器（Refiner）与 Meta-Prompt

这是整个系统的核心。它接收经过第二层打标和初筛的文本片段，利用大模型强大的上下文理解和泛化能力，进行最终的**结构化知识沉淀**。

以下是该模块运行的核心 **Meta-Prompt（提示词设计）** 细节。为了确保输出的绝对稳定，它采用了严格的格式约束和 Few-Shot 示例：

```markdown
# Role
你是一个高精度的元数据提取与知识合并专家。你的任务是分析精简后的用户对话片段，提取出用户的“长效特异性画像”和“硬核技术/业务资产”。

# Guiding Principles (过滤与合并法则)
1. 遗忘非特异性信息：忽略所有大盘研发通用的概念（如“会用 Git”、“用 IDE 写代码”、“注重代码质量”）。只保留能够唯一标识该用户技术栈和业务深度的信息。
2. 实体关系聚合：如果发现新提及的实体与已有记忆冲突，执行更新逻辑。
3. 绝对禁止幻觉：只有用户显式断言、提供、或高频深度讨论的实体才能被记录。禁止基于一句话猜测用户的职业。

# Output JSON Schema
你必须输出且仅输出一个合法的 JSON 对象，格式如下：
{
  "profile_updates": {
    "core_role": "字符串或 null，仅在有明确变更时提供",
    "infrastructure_stack": ["数据库、中间件、底层框架等核心组件，仅限 Perennial 级别"],
    "active_projects": [
      {
        "project_name": "项目或 PIP 名称",
        "tech_keywords": ["该项目涉及的核心技术"],
        "business_domain": "该项目解决的业务问题，如 ERP 返利核销",
        "lifecycle": "ACTIVE 或 HISTORICAL"
      }
    ],
    "explicit_preferences": ["用户明确提及的开发工具、IDE、或强烈的主观技术审美"]
  },
  "correction_ledger": [
    {
      "entity": "被纠正的实体名",
      "incorrect_value": "AI 之前记错的错误值",
      "correct_value": "用户纠正的绝对正确值",
      "reason": "用户纠正时的核心诉求"
    }
  ]
}

# Few-Shot Examples
Input: 
"用户：这报错是因为我用的是 OceanBase 数据库，不是标准的 SQL Server，你把刚才的 SQL 语法针对 OB 优化一下。另外，我最近在死磕 pulsar-agent 这个项目的内存管理，排查好几天了。"
Output:
```json
{
  "profile_updates": {
    "core_role": null,
    "infrastructure_stack": ["OceanBase"],
    "active_projects": [
      {
        "project_name": "pulsar-agent",
        "tech_keywords": ["内存管理"],
        "business_domain": "智能体/基础架构组件",
        "lifecycle": "ACTIVE"
      }
    ],
    "explicit_preferences": []
  },
  "correction_ledger": [
    {
      "entity": "Database",

# 特定业务文档的记忆系统
针对特定业务文档（Business Documents）的记忆系统，其设计逻辑与“用户画像记忆”有着本质的区别。

用户画像记忆追求的是“高度提炼（Summary）”**，丢弃 95% 的细节，只留几个 Key-Value 标签。而特定业务文档记忆追求的是**“高保真（High-Fidelity）、强关联、可溯源（Provenance）”。你不能随意丢弃细节，因为业务文档里的一句“返利比例为 1.2%”，丢了就是重大的业务事故。

如果在你的 `seahorse-agent` 中，核心目标是让 Agent 精准记住并理解**特定业务文档（如 API 规范、业务需求文档 PRD、财务核销对账规则）**，我建议你将信息过滤器和语义打标分类器重构为一套 **“智能文档摄取与语义分片流水线（Intelligent Document Ingestion Pipeline）”**。

以下是针对业务文档的硬核系统设计：

---

## 一、 针对业务文档的“过滤器”设计：从“噪音过滤”转为“结构解析器”

在业务文档场景下，过滤器不应该直接“扔掉文本”，而是要过滤掉**排版噪音**（如 PDF 转文本后的页眉页脚、HTML 标签、无意义的空格换行），并将文档格式化为**语义树（Semantic Tree）**。

### 1. 结构化过滤器（Layout-Aware Filter）

* **核心任务：** 提取文档的骨架。利用基于规则或轻量级视觉模型的解析器，识别出 Markdown/PDF 中的 `H1, H2, H3` 标题层级、表格（Table）和代码块（Code Block）。
* **过滤规则：**
* **提取表格（Crucial）：** 业务文档的核心数据通常在表格里（比如费率表、API 参数表）。过滤器必须将表格提取并转化为标准的 **Markdown Table** 或 **JSON 字符串**，绝对不能把表格拉平成无序的纯文本，否则大模型会彻底丧失空间对齐能力，导致算错数据。
* **拦截元数据：** 过滤掉修订日志、目录、页码、文档审批人等与核心业务逻辑无关的文本。



---

## 二、 针对业务文档的“语义打标分类器”：三维标签体系

当业务文档被切成多个语义片段（Chunks）时，分类器（通常基于微调的 Embedding 模型或轻量级分类模型）必须为每个 Chunk 打上以下三个维度的标签，以便后续精准检索：

```
                    ┌─────────────────────────┐
                    │      业务文档 Chunk      │
                    └────────────┬────────────┘
                                 │
         ┌───────────────────────┼───────────────────────┐
         ▼                       ▼                       ▼
【 1. 业务域标签 (Domain) 】  【 2. 文本类型标签 (Type) 】 【 3. 时效/状态标签 (State) 】
   - 销售返利 (Rebate)          - 约束规则 (Rule/Policy)    - 现行有效 (Active)
   - 供应链采购 (Procurement)   - 接口定义 (API_Spec)       - 历史废弃 (Deprecated)
   - 财税逻辑 (Taxation)        - 实体/名词解释 (Glossary)   - 待评审草案 (Draft)

```

### 为什么这个分类器对业务文档至关重要？

在业务场景下，如果你问：*“今年销售返利怎么核算？”*
如果没有第二层分类器，纯向量检索可能会把你去年、前年的旧文档，或者供应链的采购返利文档（因为语义极度相似）一起召回，导致 Agent 给出过期的、错误的业务结论。
**分类器的作用，就是在后台为每个 Chunk 自动加上元数据过滤网（Metadata Filter）。**

---

## 三、 第三层：业务提炼器（Refiner）的 Meta-Prompt 设计

针对业务文档，大模型提炼器的任务不是总结，而是“语义增强（Semantic Enhancement）”。

因为业务文档的 Chunk 往往缺乏上下文。例如一个 Chunk 只有一句话：*“若超期 30 天未对账，则扣除 5% 保证金。”*
如果直接存入向量库，当用户搜 *“采购对账有什么违约惩罚？”* 时，这个 Chunk 可能会因为缺少“采购对账”和“违约惩罚”这两个显式词汇而无法被检索到。

提炼器的 Meta-Prompt 应该这样设计，让 LLM 为每个 Chunk 自动生成**上下文实体补全（Parent-Child Context）**和**假设问题生成（Hypothetical Questions）**：

```markdown
# Role
你是一个资深的业务架构师。你的任务是对输入的业务文档片段（Chunk）进行语义增强，提取其中的核心业务规则，并补充上下文，以便于后续的高精度检索。

# Context Layer (当前片段的上下文骨架)
- 主文档标题：[INSERT_DOC_TITLE]
- 所在章节层级：[INSERT_CHAPTER_PATH (例如：第三章 -> 销售返利核销 -> 违约责任)]

# Input Chunk
[INSERT_ORIGINAL_CHUNK_HERE]

# Task
分析上述 Chunk，并输出一个结构化的 JSON 对象，包含以下增强字段：
1. business_rules: 提取出该片段包含的显式业务逻辑（如果-那么-否则 关系）。
2. entities_mentioned: 片段中涉及的核心业务对象（如：客户、供应商、返利比例）。
3. hypothetical_queries: 基于这个片段，业务人员可能会提出哪 3 个具体的业务问题？（用于提升向量检索的召回率）。
4. contextualized_text: 将上下文骨架融入该片段后，重写一段“自包含（Self-contained）”的完整业务描述。

# Output Format
```json
{
  "business_rules": [],
  "entities_mentioned": [],
  "hypothetical_queries": [],
  "contextualized_text": ""
}

```

```

1.  **别用纯向量存 API 文档：** API 文档（如 `getUserInfo(String userId, int type)`）对大小写、下划线、参数类型极其敏感。向量化后，这些精确的信息会被模糊掉。API 和字段的记忆，必须依靠 **BM25（如 Elasticsearch/Lucene）** 做精确匹配，或者用图数据库把 API、参数、返回值连成关系图。

# 双轨异构记忆的系统设计

## 一、 双轨异构记忆的系统架构设计

不能把“个人画像”和“业务文档”混在同一个数据库或同一套 Pipeline 里。它们的数据结构、更新频率和检索策略完全不同。你需要设计一个中央记忆路由器（Central Memory Router）来居中调度。

```
                              [ 用户输入 (Prompt) ]
                                       │
                                       ▼
                     ┌───────────────────────────────────┐
                     │ 中央记忆路由器 (Memory Router)    │
                     └─────────────────┬─────────────────┘
                                       │
         ┌─────────────────────────────┴─────────────────────────────┐
         ▼                                                           ▼
┌─────────────────────────────────┐                         ┌─────────────────────────────────┐
│     第一轨：个人画像记忆流      │                         │    第二轨：业务文档知识流       │
│    (User Profile Track)         │                         │    (Business Doc Track)         │
├─────────────────────────────────┤                         ├─────────────────────────────────┤
│ - 特点: 核心标签、高频动态更新  │                         │ - 特点: 结构化长文本、高保真    │
│ - 存储: KV 数据库 / 图数据库    │                         │ - 存储: 向量 DB + 关系型 JSON   │
│ - 触发: 显式/隐式情感与身份触发 │                         │ - 触发: 实体/关键词/向量混合检索│
└─────────────────────────────────┘                         └─────────────────────────────────┘
         │                                                           │
         └─────────────────────────────┬─────────────────────────────┘
                                       ▼
                     ┌───────────────────────────────────┐
                     │ 统一上下文组装器 (Context Weaver) │
                     └─────────────────┬─────────────────┘
                                       │ (注入 System Prompt)
                                       ▼
                             [ 大模型推理与生成 ]

```

---

## 二、 核心组件设计：如何让两者交织

### 1. 中央记忆路由器（Memory Router）的设计

当用户输入一句话时，路由器利用一个非常轻量级的意图识别分类器（可以是本地微调的 BERT，或者轻量 LLM），判断需要激活哪部分的记忆：

* **Case A（纯技术/工具讨论）：** 只激活“个人画像”，了解用户的技术偏好。
* **Case B（询问业务规则）：** 只激活“业务文档”，查找相关的条款。
* **Case C（带入个人身份解决业务问题）：** **两轨同时激活（最强形态）**。

### 2. 统一上下文组装器（Context Weaver）的 Prompt 模版

当两轨记忆同时被唤醒时，如何把它们优雅地喂给大模型？你需要使用分区隔离语法（XML Tags）来防止大模型产生注意力混淆：

```markdown
# Role
你是一个深度嵌入企业业务流的硬核 AI 架构专家搭档。请结合【当前用户信息】和【企业业务核心规范】，回答用户的问题。

# ZONE 1: 当前用户信息 (User Persona Context)
<user_profile>
- 核心角色: [从第一轨拉取的 KV：高级 Java 开发工程师]
- 技术栈资产: [OceanBase, Apache Pulsar, Trae IDE]
- 正在推进的项目: [seahorse-agent, ApiHub]
</user_profile>

# ZONE 2: 企业业务规范限制 (Business Policy Context)
<business_rules>
[从第二轨 RAG 召回并经过语义增强的 JSON 或文本片段]
- 规则源: 《集团季度末返利核销管理规范 V2》
- 核心断言: 季度末返利核销对账中，若累计差异金额 > 5000 元，付款流转将自动进入冻结状态，必须由事业部财务一号位审批方可解冻。
</business_rules>

# ZONE 3: 当前对话上下文与用户输入
[Short-term Memory Window]
用户：我刚才查了一下，这批数据的尾差累计到 5200 块了，在系统里直接改报错了，你帮我看看。

# 协同推理要求
1. 语言风格：用户是资深开发者，请直接使用硬核的技术和业务语言，拒绝废话。
2. 业务对齐：结合 ZONE 2 的规则，5200 元已经超过了 5000 元的阈值，告诉他为什么在系统里直接改会报错（因为触发了硬性冻结机制）。
3. 行动建议：结合他的技术栈（Java/OceanBase），提示他去检查哪个服务节点的审批流状态。

```

## 三、 给 `seahorse-agent` 的 Java 工程落地指南

作为一名 Java 开发者，在落地这套双轨记忆系统时，我建议你充分利用 Java 的多线程和强类型特性，做如下架构设计：

### 1. 并行检索（Parallel Retrieval Task）

当路由器决定双轨同时激活时，不要串行查询。使用 Java 的 `CompletableFuture` 并行去查两个库，然后利用 `CompletableFuture.allOf()` 进行流式聚合，将 RT（响应时间）降到最低。

```java
// 伪代码示例
CompletableFuture<UserProfile> profileFuture = CompletableFuture.supplyAsync(() -> profileRepository.getById(userId));
CompletableFuture<List<BusinessRule>> rulesFuture = CompletableFuture.supplyAsync(() -> vectorSearchService.search(query, metadataFilter));

CompletableFuture.allOf(profileFuture, rulesFuture).join();
UserProfile profile = profileFuture.get();
List<BusinessRule> rules = rulesFuture.get();

```

### 2. 数据存储的精妙配合

* **个人画像（KV + Status）：** 建议使用 **Redis** 作为热缓存（存储当前 Active 的画像），使用 **OceanBase / MySQL** 存储历史画像快照。
* **业务文档（Dense + Sparse）：** 推荐使用 **Milvus** 存储子片的向量，但把提炼出的结构化 `business_assertions` JSON 直接作为属性存入向量的 Payload 中，或者存入关系型数据库。通过一条 `doc_id` 或者是 `rule_id` 的外键完成逻辑关联。

### 3. 记忆的闭环演进（Evolution Loop）

最酷的地方在这里：**业务文档的记忆，可以随着个人画像的深入而自动精炼。**
如果个人画像里记录了该用户最近 90% 的时间都在处理“销售返利”模块。那么当他上传一份长达 10 万字的复合型企业综合制度时，你的**信息过滤器**可以自动给分类器加权，优先甚至加深去提炼和打标该文档中关于“返利”的条款，从而实现**个性化的文档摄取**。
