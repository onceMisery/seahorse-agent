# 当前性能基准说明

本文档是当前仓库的性能基准入口。早期 RepoWiki 文档中提到的 `docs/performance/rag-after-*.json` 阶段性样例文件已经不随仓库维护，因此不要再把这些文件当作可直接引用的真实基线。

## 当前事实

- 运行时观测以 Micrometer、Prometheus、Grafana 和管理端仪表盘为主要入口；完整部署见 [docker-compose.full.yml](../../docker-compose.full.yml)。
- RAG 链路的关键证据来自请求追踪、管理端仪表盘、后端指标和压测导出，而不是静态 JSON 占位样例。
- 轻量部署默认使用 `noop` 向量适配器，适合功能冒烟；完整部署使用 Milvus、Redis、Pulsar、Elasticsearch、MinIO、Ollama、Prometheus 和 Grafana，适合性能与容量验证。

## 建议基线

每次建立或刷新性能基线时，至少记录以下指标：

- 聊天首 token 延迟、完整响应延迟、错误率和取消成功率。
- 检索耗时、召回数量、重排耗时、空召回比例和知识库过滤条件。
- 记忆加载、画像写入、记忆治理任务耗时和待审核队列规模。
- 文档摄取耗时、分块数量、向量写入耗时和失败重试次数。
- JVM、数据库、向量库、消息队列、对象存储和模型服务的资源指标。

## 维护规则

- 新的性能结果应由真实压测或 E2E 流程产生，并附带测试环境、数据规模、并发设置、模型配置和部署模式。
- 如果需要重新引入机器可读基线文件，请使用明确日期或版本命名，例如 `rag-baseline-2026-06-full.yml`，并在本文档登记用途。
- 历史文档中引用 `rag-after-*.json` 的地方只表示“阶段性性能对比思路”，不表示这些文件仍存在。
