## 4. Starter 依赖治理收口设计

### 4.1 当前问题

当前 Maven 坐标名已经出现了 `starter-core` 和 `starter-all`，但依赖边界还没有真正收敛：

- `seahorse-agent-spring-boot-starter-core` 只依赖 `seahorse-agent-spring-boot-starter`，不是一个真实轻量入口。
- `seahorse-agent-spring-boot-starter` 仍声明 kernel、web、JDBC、本地/noop adapter，以及多个 optional 重型 adapter。
- `seahorse-agent-spring-boot-starter-all` 聚合官方重型 adapter。
- `seahorse-agent-bootstrap` 同时依赖 `starter-core` 和 `starter-all`，因此无法验证 core-only 启动边界。

### 4.2 目标拓扑

目标是让三个坐标语义稳定：

| 坐标 | 目标职责 | 允许依赖 |
| --- | --- | --- |
| `seahorse-agent-spring-boot-starter-core` | 默认轻量入口，可用于本地开发、测试和只需要内置 adapter 的应用 | kernel、web、repository-jdbc、mq-direct、cache-local、storage-local、vector-noop、observation-noop、Spring 自动配置 |
| `seahorse-agent-spring-boot-starter` | 兼容旧坐标，过渡期指向 core | 只依赖 `starter-core`，保留一轮兼容说明 |
| `seahorse-agent-spring-boot-starter-all` | 官方全量 adapter 聚合入口 | `starter-core` 加 OpenAI-compatible、MCP HTTP、OpenAPI、Tika、Feishu、Milvus、PGVector、Redis、S3、Pulsar、Micrometer、Elasticsearch、Lucene 等 adapter |
| `seahorse-agent-bootstrap` | 本仓库的可执行应用 | `starter-core` 加当前应用明确需要的 adapter；不再靠 `starter-all` 隐式带入全部能力 |

### 4.3 迁移步骤

1. **翻转 starter 依赖方向**
   - 将自动配置类、`META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports` 和默认 `application.properties` 迁移到 `starter-core`，或新增内部 `seahorse-agent-spring-boot-autoconfigure` 模块供 `starter-core` 依赖。
   - 将旧 `starter` 改为依赖 `starter-core` 的兼容 alias，避免 `core -> starter -> optional heavy adapter` 的语义混淆。

2. **重建 `starter-all`**
   - 让 `starter-all` 只依赖 `starter-core` 和官方重型 adapter。
   - 不在 `starter-core` 中声明 Milvus、PGVector、Redis、S3、Pulsar、OpenAI-compatible、Elasticsearch、Lucene、MCP HTTP 等外部 SDK 模块。

3. **迁移 bootstrap**
   - 删除 `bootstrap` 对 `starter-all` 的直接依赖。
   - 按 `application.properties` 中实际启用的 adapter 显式添加模块。例如生产默认需要 Redis、PGVector、OpenAI-compatible 时，就在 `bootstrap/pom.xml` 中列出这些 adapter。
   - 如果开发环境需要全量体验，新增 profile 或文档说明如何把 `starter-all` 加回应用层，而不是让 bootstrap 默认全量依赖。

4. **增加依赖边界回归检查**
   - 为 `starter-core` 增加依赖树快照或 Maven Enforcer 规则，禁止引入外部服务 SDK。
   - 为 `starter-all` 增加 smoke test，确保全量 adapter 仍能被 Spring 条件装配发现。
   - 为 `bootstrap` 增加依赖树检查，确认重型 adapter 都来自显式依赖。

### 4.4 验收标准

- `mvn dependency:tree -pl seahorse-agent-spring-boot-starter-core` 不出现 Milvus、PGVector、Elasticsearch、Lucene、Pulsar、S3、OpenAI-compatible、MCP HTTP、Feishu、Redis 等重型 adapter。
- `mvn dependency:tree -pl seahorse-agent-bootstrap` 中的重型 adapter 均能在 `bootstrap/pom.xml` 找到显式声明。
- `starter` 不再是 core 的上游依赖。
- core-only Spring context 能启动并提供 noop/local/direct/JDBC 默认能力。
- all-starter Spring context 能在相关类存在时加载官方重型 adapter 自动配置。

## 5. 元数据治理 JDBC 端口拆分设计

### 5.1 当前问题

`JdbcMetadataGovernanceRepositoryAdapter` 已经抽出了 JSON、schema usage、canonical write、review、quarantine、backfill、quality report 等 support，但主类仍承担兼容门面，并且 `SeahorseAgentMetadataAdapterAutoConfiguration` 仍把同一个 adapter 暴露为多个 metadata port。

这会带来三个问题：

- 单个类仍知道过多表结构、row mapper 和统计逻辑。
- 使用一个小端口的调用方仍被绑定到大 adapter。
- 后续新增 metadata 子域时容易继续修改同一个主类和同一个自动配置入口。

### 5.2 子域 Bean 切分

建议保留一轮兼容门面，但默认 Bean 改为细粒度 adapter：

| 子域 | 新 adapter 建议 | 暴露端口 |
| --- | --- | --- |
| Schema | `JdbcMetadataSchemaRepositoryAdapter` | `MetadataSchemaRegistryPort`、`MetadataSchemaManagementRepositoryPort`、`MetadataSchemaIndexStatusPort` |
| Dictionary | `JdbcMetadataDictionaryRepositoryAdapter` | `MetadataDictionaryPort`、`MetadataDictionaryManagementRepositoryPort` |
| Extraction result | `JdbcMetadataExtractionResultRepositoryAdapter` | `MetadataExtractionResultRepositoryPort`、`MetadataExtractionResultManagementRepositoryPort` |
| Review | `JdbcMetadataReviewRepositoryAdapter` | `MetadataReviewQueuePort`、`MetadataReviewManagementRepositoryPort` |
| Quarantine | `JdbcMetadataQuarantineRepositoryAdapter` | `MetadataQuarantinePort`、`MetadataQuarantineManagementRepositoryPort` |
| Canonical write | `JdbcMetadataCanonicalWriteRepositoryAdapter` | `MetadataCanonicalWritePort` |
| Backfill | `JdbcMetadataBackfillJobRepositoryAdapter` | `MetadataBackfillJobRepositoryPort` |
| Quality report | `JdbcMetadataQualityReportRepositoryAdapter` | `MetadataQualityReportRepositoryPort` |
| Schema usage report | `JdbcMetadataSchemaUsageReportRepositoryAdapter` | `MetadataSchemaUsageReportRepositoryPort` |

`JdbcMetadataJsonSupport`、`JdbcMetadataColumnDetector` 等保持共享 utility，但不再让一个 adapter 直接持有所有 SQL。

### 5.3 事务边界

| 子域 | 事务边界 | 说明 |
| --- | --- | --- |
| Schema/Dictionary | 单命令事务 | 创建、更新、启停、删除各自独立；读路径不持事务。 |
| Extraction result | 读路径独立，写路径跟随 ingestion 事务 | 管理分页与详情查询不应影响入库主链路。 |
| Review | decision + audit 同事务 | approve/reject/modify 必须和审核审计记录一起提交。 |
| Quarantine | 状态迁移同事务 | release/retry/ignore 与状态、更新时间、操作者信息一起提交。 |
| Canonical write | canonical metadata + usage/index status 同事务或 outbox | 如果写入后需要异步索引同步，事务内只写 outbox，不直接调用外部搜索 SDK。 |
| Backfill | job claim 与状态更新小事务 | 扫描、LLM、重抽取等长耗时步骤不能包在一个数据库事务里。 |
| Quality report | snapshot append 独立事务 | 报表计算可以读多表，但落库是一次快照写入。 |

### 5.4 自动配置迁移

1. `SeahorseAgentMetadataAdapterAutoConfiguration` 先创建共享 `JdbcTemplate`、`JdbcMetadataJsonSupport`。
2. 每个子域 adapter 用自己的 `@ConditionalOnMissingBean(Port.class)` 暴露端口。
3. 兼容门面 `JdbcMetadataGovernanceRepositoryAdapter` 改为可选 Bean，只在旧端口组合被显式请求或兼容开关打开时创建。
4. 门面内部只委托到细粒度 port，不再持有 SQL、row mapper 或统计逻辑。

### 5.5 验收标准

- 默认 Spring context 中，metadata 端口不再全部指向同一个 `JdbcMetadataGovernanceRepositoryAdapter` 实例。
- `JdbcMetadataGovernanceRepositoryAdapter` 被标记为兼容门面，且方法体只委托。
- 每个子域 adapter 有独立单元测试和至少一个自动配置覆盖测试。
- 原有 controller/service 的注入端口不变。
- 旧数据库缺列兼容逻辑仍在子域 adapter 内保留，不因拆分类而丢失。

## 6. 术语映射在线扩展设计

### 6.1 当前问题

术语映射管理链路已经存在：

- 前端 `/admin/mappings` 能管理 source term、target term、match type、priority、enabled。
- `QueryTermMappingRepositoryPort` 和 `JdbcQueryTermMappingRepositoryAdapter` 已读写 `t_query_term_mapping`。
- `KernelQueryTermMappingService` 在新增、更新、删除时清理缓存 key。

但在线查询链路还没有真正消费这些配置：

- `QueryTermExpansionPort` 只有 noop。
- `RuleBasedQueryOptimizerPort` 会调用 `QueryTermExpansionPort.expand()`，但当前没有生产实现。
- `RuleBasedQueryOptimizerPort` 将 `expandedTerms` 写入 `QueryOptimizationResult` 后，只通过 debug log 输出；下游 rewrite/retrieval 尚未消费扩展词。

因此，本缺口应拆成两个阶段：先让端口能从 JDBC 返回扩展词，再让扩展词进入检索和改写链路。

### 6.2 JDBC adapter

新增 adapter：

```text
seahorse-agent-adapter-repository-jdbc/
  src/main/java/com/miracle/ai/seahorse/agent/adapters/repository/jdbc/JdbcQueryTermExpansionAdapter.java
```

职责：

- 读取 `t_query_term_mapping` 中 `deleted = 0 and enabled = 1` 的规则。
- 按 `priority ASC, update_time DESC` 排序。
- 对输入 query 做匹配，返回 `Map<sourceTerm, List<targetTerm>>`。
- 使用现有缓存 key `seahorse-agent:query-term:mappings`，复用管理端 CRUD 的失效行为。

匹配语义：

| `match_type` | 语义 | 兼容说明 |
| --- | --- | --- |
| `1` | 精确文本匹配。query 中出现完整 source term 即命中；ASCII 字符大小写不敏感，中文按原文匹配。 | 对应当前前端“精确匹配”。 |
| `2` | 前缀/模糊匹配。query 或 token 以前缀命中 source term，旧库注释中的“模糊”也按此类处理。 | 兼容旧 DB 注释“1 精确 2 模糊”和当前前端“前缀匹配”。 |
| `3` | 正则匹配。source term 作为正则表达式匹配 query。 | 需要开关和安全限制，默认可先关闭。 |

安全规则：

- 单条 source term 长度超过配置上限时跳过。
- 正则规则需要配置 `seahorse-agent.query-term-expansion.regex-enabled=true` 才启用。
- 正则编译失败只跳过该规则并记录 debug/trace，不影响主查询。
- 单次查询最多返回配置上限数量的扩展词，避免 prompt 和检索 query 膨胀。

### 6.3 自动配置

在 JDBC repository 自动配置中新增：

- `@ConditionalOnBean(DataSource.class)`
- `@ConditionalOnProperty(prefix = "seahorse-agent.query-term-expansion", name = "enabled", havingValue = "true", matchIfMissing = true)`
- `@ConditionalOnMissingBean(QueryTermExpansionPort.class)`

如果没有 JDBC adapter，保留 `QueryTermExpansionPort.noop()` 的降级行为。

### 6.4 下游消费

阶段一只让 `RuleBasedQueryOptimizerPort` 产生真实 `expandedTerms`；阶段二让扩展词真正影响召回：

1. `KernelChatPreparationSupport.optimizeQuery()` 保持把 `QueryOptimizationResult` 写入 `StreamChatContext`。
2. `resolveRewriteInput()` 仍只把 `optimizedQuestion` 传给 `QueryRewritePort`，不直接拼接扩展词，避免污染用户原问题。
3. 检索前新增一个轻量 resolver，把 `expandedTerms` 写入 `SearchContext.metadata` 或 `RetrievalOptions.channelSettings`。
4. `KeywordSearchChannelFeature` 优先消费扩展词，把关键词查询扩展为 `original OR expanded`。
5. 向量通道默认不拼接扩展词；如需使用，应通过配置启用“扩展 query variant”，单独生成一个辅助向量候选，避免改变原始语义。
6. Prompt 组装不直接展示扩展词，除非 trace/debug 模式需要解释检索命中。

### 6.5 观测和回退

- trace metadata 记录 `appliedRules=term_expansion`、命中的 source term 和扩展词数量。
- 缓存加载失败、DB 查询失败或规则解析失败时返回空 map，不中断主聊天链路。
- 提供配置开关 `seahorse-agent.query-term-expansion.enabled=false` 可整体关闭。
- 提供配置 `seahorse-agent.query-term-expansion.max-rules` 和 `max-expanded-terms` 控制成本。

### 6.6 验收标准

- 新增 `JdbcQueryTermExpansionAdapterTests` 覆盖 exact、prefix/fuzzy、regex disabled、disabled/deleted ignored、priority ordering、duplicate target 去重。
- 新增自动配置测试证明 JDBC 环境下 `QueryTermExpansionPort` 不再是 noop。
- 新增 `RuleBasedQueryOptimizerPort` 测试证明匹配到的 source term 会进入 protected terms，target term 会进入 expanded terms。
- 新增检索链路测试证明 keyword channel 能消费 expanded terms，而原始问题仍写入 memory 和 prompt 上下文。
- `/admin/mappings` 创建、更新、删除后，缓存失效并在下一次查询重新加载规则。
