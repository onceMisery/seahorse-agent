## 4. Starter 依赖治理当前状态

### 4.1 已完成收口

当前 Maven 坐标语义已经完成收口：

- `seahorse-agent-spring-boot-autoconfigure` 承载 Spring Boot 自动配置类、`META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports`、默认 `application.properties` 和内置 skills 资源。
- `seahorse-agent-spring-boot-starter-core` 只依赖 `seahorse-agent-spring-boot-autoconfigure`，作为默认轻量入口。
- `seahorse-agent-spring-boot-starter` 已改为只依赖 `starter-core` 的兼容 alias，不再是 core 的上游依赖。
- `seahorse-agent-spring-boot-starter-all` 依赖 `starter-core`，再聚合官方重型 adapter，并已有 classpath/自动配置候选冒烟测试。
- `seahorse-agent-bootstrap` 默认只依赖 `starter-core` 和应用层显式运行库，不再依赖 `starter-all`；`bootstrap/pom.xml` 通过 Maven Enforcer 禁止回退到聚合 starter。

### 4.2 目标拓扑

目标是让三个坐标语义稳定：

| 坐标 | 目标职责 | 允许依赖 |
| --- | --- | --- |
| `seahorse-agent-spring-boot-starter-core` | 默认轻量入口，可用于本地开发、测试和只需要内置 adapter 的应用 | kernel、web、repository-jdbc、mq-direct、cache-local、storage-local、vector-noop、observation-noop、Spring 自动配置 |
| `seahorse-agent-spring-boot-starter` | 兼容旧坐标，过渡期指向 core | 只依赖 `starter-core`，保留一轮兼容说明 |
| `seahorse-agent-spring-boot-starter-all` | 官方全量 adapter 聚合入口 | `starter-core` 加 OpenAI-compatible、MCP HTTP、OpenAPI、Tika、Feishu、Milvus、PGVector、Redis、S3、Pulsar、Micrometer、Elasticsearch、Lucene 等 adapter |
| `seahorse-agent-bootstrap` | 本仓库的可执行应用 | `starter-core` 加当前应用明确需要的 adapter；不再靠 `starter-all` 隐式带入全部能力 |

### 4.3 迁移步骤

1. **翻转 starter 依赖方向**
   - 已完成：新增内部 `seahorse-agent-spring-boot-autoconfigure` 模块，承载自动配置类、`META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports` 和默认 `application.properties`。
   - 已完成：`starter-core` 依赖 `autoconfigure`，旧 `starter` 改为依赖 `starter-core` 的兼容 alias，避免 `core -> starter -> optional heavy adapter` 的语义混淆。

2. **重建 `starter-all`**
   - 已完成：`starter-all` 只依赖 `starter-core` 和官方重型 adapter。
   - 不在 `starter-core` 中声明 Milvus、PGVector、Redis、S3、Pulsar、OpenAI-compatible、Elasticsearch、Lucene、MCP HTTP 等外部 SDK 模块。

3. **迁移 bootstrap**
   - 已完成：删除 `bootstrap` 对 `starter-all` 的直接依赖。
   - 按 `application.properties` 中实际启用的 adapter 显式添加模块。例如生产默认需要 Redis、PGVector、OpenAI-compatible 时，就在 `bootstrap/pom.xml` 中列出这些 adapter。
   - 如果开发环境需要全量体验，新增 profile 或文档说明如何把 `starter-all` 加回应用层，而不是让 bootstrap 默认全量依赖。

4. **增加依赖边界回归检查**
   - 已完成：为 `starter-core` 增加 Maven Enforcer 规则，禁止引入 `starter-all` 和官方重型 adapter。
   - 已完成：为 `starter-core` 增加 core-only context 测试，验证 local/direct/noop/JDBC 默认能力，并确认官方重型 adapter 不在测试类路径中。
   - 已完成：为 `starter-all` 增加 smoke test，确保全量 adapter 类和相关 Spring Boot 自动配置候选仍能被全量 starter 测试类路径发现。
   - 已完成：为 `bootstrap` 增加 Maven Enforcer 规则，禁止默认应用依赖 `starter-all`。

### 4.4 验收标准

## 5. 元数据治理 JDBC 端口拆分当前状态

### 5.1 已完成收口

`JdbcMetadataGovernanceRepositoryAdapter` 已经抽出了 JSON、schema usage、canonical write、review、quarantine、backfill、quality report 等 support。2026-06-08 后，默认 Spring Bean 边界和 SQL/support owner 已完成收口：

- `SeahorseAgentMetadataAdapterAutoConfiguration` 默认创建 `JdbcMetadataGovernanceRepositoryDelegate`，不再默认创建 `JdbcMetadataGovernanceRepositoryAdapter` Bean。
- metadata 出站端口默认由独立 JDBC adapter 暴露为细粒度 Bean，包括 schema、dictionary、extraction、review、quarantine、canonical write、backfill、quality report、schema usage report 等端口。
- 默认 Bean 类型分别是 `JdbcMetadataSchemaRepositoryAdapter`、`JdbcMetadataDictionaryRepositoryAdapter`、`JdbcMetadataExtractionResultRepositoryAdapter`、`JdbcMetadataReviewRepositoryAdapter`、`JdbcMetadataQuarantineRepositoryAdapter`、`JdbcMetadataCanonicalWriteRepositoryAdapter`、`JdbcMetadataBackfillJobRepositoryAdapter`、`JdbcMetadataQualityReportRepositoryAdapter` 和 `JdbcMetadataSchemaUsageReportRepositoryAdapter`。
- 旧兼容门面 `JdbcMetadataGovernanceRepositoryAdapter` 仅在显式配置 `seahorse-agent.adapters.metadata-governance-compatibility.facade-bean-enabled=true`，且 `seahorse-agent.adapters.repository.type=jdbc` 时创建。
- 旧兼容门面已改为只持有上述独立子域 adapter 并逐方法委托，不再持有 `JdbcTemplate`、SQL、row mapper 或 support owner。
- 自动配置测试 `SeahorseAgentKernelMetadataAutoConfigurationTests` 覆盖默认细粒度 Bean、显式兼容门面、非 JDBC repository 不创建兼容门面三条路径。
- `JdbcMetadataGovernanceRepositoryAdapterCompatibilityTests` 覆盖兼容门面只拥有子域 adapter 字段，防止 SQL/support owner 回流。

这意味着“默认 Spring context 中 metadata 端口不再全部指向同一个 `JdbcMetadataGovernanceRepositoryAdapter` 实例”和“兼容门面不再持有 SQL/support owner”都已经实现。

### 5.2 剩余问题

当前仍保留两个兼容边界：

- `JdbcMetadataGovernanceRepositoryAdapter` 仍作为旧调用方的一体化类型保留，但只在显式开关下创建，内部只委托到独立子域 adapter。
- `JdbcMetadataPortAdapters` 仍作为缺失端口 fallback/兼容辅助保留；默认路径在对应子域 adapter 已存在时不会使用它。

剩余风险主要是维护层面的：

- 后续新增 metadata 子域时，需要继续优先新增独立 adapter，避免把逻辑塞回兼容门面。
- `JdbcMetadataPortAdapters` 的 fallback 入口应保持薄委托，不能重新承载 SQL。
- 如果未来删除兼容门面，需要先确认外部应用没有直接注入 `JdbcMetadataGovernanceRepositoryAdapter` 类型。

### 5.4 事务边界

| 子域 | 事务边界 | 说明 |
| --- | --- | --- |
| Schema/Dictionary | 单命令事务 | 创建、更新、启停、删除各自独立；读路径不持事务。 |
| Extraction result | 读路径独立，写路径跟随 ingestion 事务 | 管理分页与详情查询不应影响入库主链路。 |
| Review | decision + audit 同事务 | approve/reject/modify 必须和审核审计记录一起提交。 |
| Quarantine | 状态迁移同事务 | release/retry/ignore 与状态、更新时间、操作者信息一起提交。 |
| Canonical write | canonical metadata + usage/index status 同事务或 outbox | 如果写入后需要异步索引同步，事务内只写 outbox，不直接调用外部搜索 SDK。 |
| Backfill | job claim 与状态更新小事务 | 扫描、LLM、重抽取等长耗时步骤不能包在一个数据库事务里。 |
| Quality report | snapshot append 独立事务 | 报表计算可以读多表，但落库是一次快照写入。 |

### 5.5 自动配置迁移

1. 当前已由 `SeahorseAgentMetadataAdapterAutoConfiguration` 为每个子域 adapter 使用自己的 `@ConditionalOnMissingBean(Port.class)` 暴露端口。
2. `JdbcMetadataGovernanceRepositoryDelegate` 和 `JdbcMetadataPortAdapters` 仅作为缺失端口 fallback/兼容辅助，默认端口 Bean 存在时不会接管执行路径。
3. 兼容门面 `JdbcMetadataGovernanceRepositoryAdapter` 已改为可选 Bean，只在兼容开关打开且 JDBC repository 启用时创建。
4. 门面内部已只委托到细粒度 adapter，不再持有 SQL、row mapper 或统计逻辑。

### 5.6 验收标准

- 已满足：默认 Spring context 中，metadata 端口不再全部指向同一个 `JdbcMetadataGovernanceRepositoryAdapter` Bean。
- 已满足：兼容门面 `JdbcMetadataGovernanceRepositoryAdapter` 只在显式兼容开关下创建。
- 已满足：`JdbcMetadataGovernanceRepositoryAdapter` 方法体只委托，SQL 和 row mapper 下沉到子域 adapter。
- 已满足：每个子域 adapter 有独立单元测试，自动配置测试覆盖默认细粒度 Bean 和兼容门面路径。
- 已满足：旧数据库缺列兼容逻辑保留在对应 support/adapter 内，不因拆分类而丢失。

### 6.2 JDBC adapter 行为

实现位置：

```text
seahorse-agent-adapter-repository-jdbc/
  src/main/java/com/miracle/ai/seahorse/agent/adapters/repository/jdbc/JdbcQueryTermExpansionAdapter.java
```

当前行为：

- 读取 `t_query_term_mapping` 中 `deleted = 0 and enabled = 1` 的规则。
- 按 `priority ASC, update_time DESC` 排序。
- 对输入 query 做匹配，返回 `Map<sourceTerm, List<targetTerm>>`。
- 使用缓存 key `seahorse-agent:query-term:mappings`，复用管理端 CRUD 的失效行为。
- DB 查询失败、缓存加载失败或规则解析失败时 fail-open 返回空 map，不中断主聊天链路。

匹配语义：

| `match_type` | 当前语义 | 说明 |
| --- | --- | --- |
| `1` | 精确包含匹配，使用标准化后的小写文本比较。 | 对应当前前端“精确匹配”。 |
| `2` | 前缀/模糊匹配，兼容旧库注释中的“模糊”和当前前端“前缀匹配”。 | 用于较宽松的术语命中。 |
| `3` | 正则匹配。 | 默认关闭，需配置 `seahorse-agent.query-term-expansion.regex-enabled=true`。 |

成本和安全配置：

- `seahorse-agent.query-term-expansion.enabled`：默认 `true`。
- `seahorse-agent.query-term-expansion.regex-enabled`：默认 `false`。
- `seahorse-agent.query-term-expansion.max-rules`：限制单次加载规则数。
- `seahorse-agent.query-term-expansion.max-expanded-terms`：限制单次查询返回扩展词数。
- `seahorse-agent.query-term-expansion.max-source-term-length`：跳过过长 source term。
- `seahorse-agent.query-term-expansion.cache-ttl`：控制规则缓存 TTL。

### 6.3 下游消费边界

扩展词只影响 keyword 检索，不直接改写用户问题：

1. `resolveRewriteInput()` 仍只把 `optimizedQuestion` 传给 `QueryRewritePort`。
2. 原始问题仍作为用户输入和向量语义检索的主语义来源。
3. `KeywordSearchChannelFeature` 拼接主问题和扩展词，交给 keyword repository 适配器处理。
4. Prompt 组装不直接展示扩展词，除非 trace/debug 后续需要解释检索命中。

这种边界能让术语映射提升关键词召回，同时避免扩展词污染向量查询、改写问题或用户可见上下文。

### 6.4 已有验证

- `JdbcQueryTermExpansionAdapterTests` 覆盖 exact、prefix/fuzzy、regex disabled、disabled/deleted ignored、priority ordering、duplicate target 去重。
- `SeahorseAgentOpsRepositoryAutoConfigurationTests` 覆盖 JDBC 自动配置下的 `QueryTermExpansionPort` 装配。
- `KeywordSearchChannelFeatureTests` 覆盖 keyword channel 消费扩展词且不修改主问题。

### 6.5 后续可选增强

- 如后续引入多租户或 domain-aware 术语映射，需要在 adapter 查询条件和缓存 key 中加入租户/领域维度。
